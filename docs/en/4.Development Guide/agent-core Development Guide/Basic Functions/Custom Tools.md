openJiuwen supports the following three ways to convert developer-defined tools into tools that can be recognized and invoked by LLMs:

- Toolifying local functions: Provides the `@tool` decorator, which can register local Python functions as a Tool class recognizable by LLMs, enabling Function Calling-like capabilities.
- Wrapping Restful interfaces: Provides the `RestfulApi` class to conveniently define and manage Restful-style API interfaces, including interface name, description, parameters, path, headers, request method, and response information, enabling unified and standardized interface management and invocation.
- Wrapping MCP interfaces: Provides the `MCPTool` class to conveniently wrap MCP services, enabling unified interface management and invocation of MCP services.

# Toolifying Local Functions

openJiuwen provides the `@tool` decorator to wrap local Python functions into a `Tool` class, making them recognizable and callable by LLMs.

## Using the @tool decorator to wrap local functions

Developer-defined local Python functions can be wrapped using the `@tool` decorator. The `@tool` decorator requires the developer to configure the function name, description, and parameters to accurately describe the local Python function. It is recommended to refer to the local function's definition when configuring.

```python
from openjiuwen.core.foundation.tool import tool

@tool(
    name="add",
    description="Local addition plugin",
    input_params={
        "type": "object",
        "properties": {
            "a": {
                "type": "integer",
                "description": "First parameter"
            },
            "b": {
                "type": "integer",
                "description": "Second parameter"
            }
        },
        "required": ["a", "b"]
    }
)
def add(a: int, b: int) -> int:
    return a + b
```

## Invoking the tool

The `Tool` class generated by the `@tool` decorator can be recognized and invoked by LLMs. Prepare the corresponding tool inputs according to the parameter definitions in the `@tool` decorator, then call the Tool class's `invoke` method to trigger the execution logic of the local function.

```python
# Generate a Tool class that can be recognized and invoked by LLMs
import asyncio
tool_info = add.card.tool_info()
print(f"Invocation successful, result: {tool_info}")

inputs = {
    "a": 5,
    "b": 1
}
result = asyncio.run(add.invoke(inputs=inputs))
print(f"Invocation successful, result: {result}")
```

Sample output:

```text
Invocation successful, result: type='function' name='add' description='Local addition plugin' parameters={'type': 'object', 'properties': {'a': {'type': 'integer', 'description': 'First parameter'}, 'b': {'type': 'integer', 'description': 'Second parameter'}}, 'required': ['a', 'b']}
Invocation successful, result: 6
```

# Encapsulating Restful APIs

openJiuwen provides the `RestfulApi` class to wrap Restful API services. The resulting `RestfulApi` class can be recognized and invoked by LLMs.

## Using the RestfulApi class to wrap Restful API services

After developers implement and deploy custom RESTful API services, they can use the `RestfulApi` class provided by openJiuwen for encapsulation. The `RestfulApi` class requires configuration of a `RestfulApiCard`, which defines the tool name, description, parameters, path, headers, request method, and response information to describe the RESTful API service. It is recommended to follow the interface definition of the RESTful API service for configuration.

```python
# Example of encapsulating a weather RESTful API service
from openjiuwen.core.foundation.tool import RestfulApi, RestfulApiCard

weather_plugin = RestfulApi(
    card=RestfulApiCard(
        id="WeatherReporter-1",
        name="WeatherReporter",
        description="Weather query plugin",
        input_params={
            "type": "object",
            "properties": {
                "location": {"description": "Location for weather query, must be in English", "type": "string"},
                "date": {"description": "Date for weather query, format: YYYY-MM-DD", "type": "string"},
            },
            "required": ["location", "date"],
        },
        url="http://127.0.0.1:8000",
        headers={},
        method="GET",
    )
)
```

## Invoking the tool

The `RestfulApi` class can be recognized and invoked by LLMs. Refer to the parameter definitions of the `RestfulApi` class, prepare the tool inputs, and calling the `RestfulApi` class's `invoke` method will run the Restful API service remotely.

```python
# RestfulApi instance, recognizable and callable by LLMs
tool_info = weather_plugin.card.tool_info()
print(f"Invocation successful, result: {tool_info}")

inputs = {
    "location": "beijing",
    "date": "2023-10-01"
}
loop = asyncio.get_event_loop()
result = loop.run_until_complete(weather_plugin.invoke(inputs=inputs))
print(f"Invocation successful, result: {result}")
```

Sample output:

```text
Invocation successful, result: type='function' name='WeatherReporter' description='Weather query plugin' parameters=Parameters(type='object', properties={'location': {'description': 'Location for weather query, must be in English', 'type': 'string'}, 'date': {'description': 'Date for weather query, format: YYYY-MM-DD', 'type': 'string'}}, required=['location', 'date'])
Invocation successful, result: The weather in beijing on 2023-10-01 is sunny
```

# Encapsulating MCP Interfaces

openJiuwen provides the `MCPTool` class to wrap MCP services. The resulting `MCPTool` class can be recognized and invoked by LLMs.

## Using the MCPTool class to wrap MCP services

After developers implement and deploy custom MCP services, they can use the `MCPTool` class provided by openJiuwen for encapsulation. The `MCPTool` class requires configuration of an `McpCard` to define the input schema, service name, and other information to describe the MCP service, along with an MCP Client. Currently, various MCP clients such as SSE, stdio, and playwright are supported.

```python
# Example of encapsulating a weather MCP service
import asyncio

from openjiuwen.core.foundation.tool import MCPTool, SseClient

mcp_client = SseClient(server_path="your weather mcp url", name="MockSseClient")
asyncio.run(mcp_client.connect(timeout=10))
tool_info_list = asyncio.run(mcp_client.list_tools())
for tool_info in tool_info_list:
    print("Invocation successful, result:", tool_info.model_dump_json())
    
mcp_tool = MCPTool(mcp_client=mcp_client, tool_info=tool_info_list[0])
```

Sample output:

```text
Invocation successful, result: {"type":"function","name":"query_weather","description":"","parameters":null,"input_schema":{"properties":{"location":{"title":"Location","type":"string"}},"required":["location"],"title":"query_weatherArguments","type":"object"},"server_name":""}
```

## Invoking the tool

The `MCPTool` class can be recognized and invoked by LLMs. Refer to the parameter definitions of the `MCPTool` class, prepare the tool inputs, and calling the `MCPTool` class's `ainvoke` method will run the MCP service remotely.

```python
# MCPTool instance, recognizable and callable by LLMs
tool_info = mcp_tool.card.tool_info()
print(f"Invocation successful, result: {tool_info}")

inputs = {
    "location": "beijing",
}
result = asyncio.run(mcp_tool.invoke(inputs=inputs))
print(f"Invocation successful, result: {result}")
asyncio.run(mcp_client.disconnect())
```

Sample output:

```text
Invocation successful, result: type='function' name='query_weather' description='' parameters=None input_schema={'properties': {'location': {'title': 'Location', 'type': 'string'}}, 'required': ['location'], 'title': 'query_weatherArguments', 'type': 'object'} server_name=''
Invocation successful, result: {'result': '{\n  "location": "beijing",\n  "temperature": "22℃",\n  "condition": "sunny"\n}'}
```
