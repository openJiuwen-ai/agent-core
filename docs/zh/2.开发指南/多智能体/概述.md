多智能体（Multi-Agent）将多个独立 Agent 协同组织在一起，通过相互通信与协作完成复杂任务的一种系统架构。相比单一 Agent，多智能体体系能够并行分工、各司其职，在处理规模较大、逻辑复杂或需要多角色配合的任务时具有明显优势。

openJiuwen 多智能体框架围绕 **TeamRuntime**（通信运行时）和 **BaseTeam**（团队封装）两个核心概念设计，既支持轻量级的直接运行时使用，也支持将完整团队对外暴露为统一接口；同时通过 **Agent as Tool** 能力，允许将任意 Agent 作为工具供其他 Agent 调用。

# 核心概念

## TeamRuntime

`TeamRuntime` 是多智能体通信的核心运行时，负责：

- **Agent 注册与管理**：以 Card + Provider 模式延迟创建 Agent 实例，统一维护 Agent 元信息。
- **消息路由**：内置 MessageBus，支持点对点（P2P）和发布-订阅（Pub-Sub）两种通信模式。
- **生命周期管理**：提供 `start()` / `stop()` 及异步上下文管理器，保障资源安全释放。

`TeamRuntime` 可单独使用，也可作为 `BaseTeam` 的通信底座。

## CommunicableAgent

`CommunicableAgent` 是一个 Mixin 类，为 Agent 提供通信能力。继承它的 Agent 可在 `invoke` 逻辑中直接调用 `send()` / `publish()` / `subscribe()` 等方法，与团队内其他 Agent 进行通信。运行时在 Agent 首次实例化时自动完成绑定，无需手动操作。

## BaseTeam

`BaseTeam` 是团队的抽象基类，采用 **Card + Config** 模式对外暴露统一接口：

- `TeamCard`：团队身份定义（id、name、description、成员列表等）。
- `TeamConfig`：运行时参数（最大 Agent 数、消息超时等）。
- 所有 Agent 管理委托给内部的 `TeamRuntime`。
- 子类实现 `invoke()` / `stream()` 即可作为完整团队对外服务。

## Agent as Tool

openJiuwen 的 `AbilityManager` 支持将 `AgentCard` 作为能力（Ability）注册。当主 Agent 决策调用某个子 Agent 时，框架自动从 ResourceManager 获取实例并执行其 `invoke()`，结果以 `ToolMessage` 形式返回，整个过程对上层完全透明。

# 通信模式

| 模式 | 方法 | 特点 |
|------|------|------|
| 点对点（P2P） | `runtime.send()` / `agent.send()` | 一对一，同步等待响应 |
| 发布-订阅（Pub-Sub） | `runtime.publish()` / `agent.publish()` | 一对多广播，Fire-and-Forget |
| 混合（Hybrid） | P2P + Pub-Sub 组合 | 灵活编排复杂协作流程 |

订阅支持精确匹配和通配符（`*`、`?`）两种方式，例如 `task_events`、`task_*`。

# 功能概览

| 功能模块 | 说明 |
|----------|------|
| TeamRuntime + CommunicableAgent | 轻量级运行时，适合直接编排多 Agent 协作 |
| BaseTeam | 团队封装，对外暴露统一 invoke/stream 接口 |
| Agent as Tool | 将 Agent 作为工具供其他 Agent 调用 |


## AgentTeams 架构（Leader-Teammate 协作）

通过 Leader（负责人）和 Teammates（队友）的协同工作完成复杂任务，支持跨进程、持久化和自动恢复。

### 核心概念

**Leader（负责人）**
- 负责任务分解、分配和协调
- 管理团队成员的生命周期
- 处理用户输入并分配给合适的队友

**Teammate（队友）**
- 执行特定领域的任务
- 通过消息系统与 Leader 通信
- 可以独立运行在单独的进程中

**架构组件**
- **Transport（传输层）**: 负责智能体间消息传递（支持 pyzmq、team_runtime）
- **Storage（存储层）**: 持久化团队状态、任务列表和消息（支持 sqlite）
- **CoordinationLoop（协调循环）**: 管理智能体的执行流程和事件处理

### 团队生命周期

| 模式 | 说明 |
|------|------|
| Temporary（临时） | 任务完成后自动解散团队，适用于一次性任务 |
| Persistent（持久） | 团队状态和成员跨会话保留，支持断点续传和崩溃恢复 |

### 队友执行模式

| 模式 | 说明 |
|------|------|
| Plan Mode | 队友完成任务需要 Leader 批准，适用于需要严格控制的场景 |
| Build Mode | 队友直接完成任务，无需批准，适用于信任队友的场景 |

### 高级特性
- **健康检查与自动恢复**: Leader 定期检查队友进程状态，检测到不健康时自动重启
- **持久待命**: 持久团队每轮结束后进入待命状态，队友进程保持存活，下一次 `invoke()` 自动恢复
- **断点续传**: 支持通过 `resume_persistent_team()`（同进程）或 `recover_agent_team()`（崩溃恢复）恢复团队状态
- **用户 @mention**: 用户可通过 `@member_id message` 语法直接向特定队友发送消息，绕过 leader
- **预定义成员**: 预配置团队成员跳过动态 spawn，自动注册到数据库
- **跨进程通信**: 通过 pyzmq 支持跨进程、跨主机协作

### 适用场景
- 需要跨进程协作的复杂任务
- 长期运行的团队项目
- 需要持久化和恢复能力的场景
- 需要健康检查和自动恢复的生产环境
