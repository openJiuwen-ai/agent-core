# 沙箱（Sandbox）

沙箱模式是 SysOperation 的一种运行模式，用于把文件系统、Shell 和代码执行等能力路由到隔离环境中执行。关于 SysOperation 的基础概念、直接调用、Tool 调用和 Agent 集成方式，参见[系统操作](./系统操作.md)。本文只介绍沙箱模式本身的设计、隔离粒度、当前使用方式以及扩展方法。

沙箱模式的核心特性包括：

- **环境隔离**：在独立的沙箱环境中执行操作，与本地系统隔离
- **无状态设计**：SysOperation 实例本身无状态，沙箱生命周期由 Gateway 统一管理
- **灵活扩展**：支持多种沙箱类型和启动方式，用户可自定义 Provider 和 Launcher
- **多粒度隔离**：支持 SYSTEM、SESSION、CUSTOM 三种隔离级别

> **说明**：当前版本支持连接已启动的 AIO 沙箱。本地运行可使用 Docker 或 轻量沙箱；大规模远程/生产部署推荐使用 Kubernetes（K8s）等云服务。

***

## 1. 核心原理

### 1.1 分层关系

沙箱模式从上到下可以分成四层：

| 层级              | 职责   | 说明                                                                            |
| --------------- | ---- | ----------------------------------------------------------------------------- |
| **Operation 层** | 用户接口 | `SandboxFsOperation`、`SandboxShellOperation`、`SandboxCodeOperation`，提供统一的操作接口 |
| **Protocol 层**  | 接口抽象 | 定义沙箱底层需要实现的最小接口集，如 `BaseFsProtocol`、`BaseShellProtocol`                       |
| **Gateway 层**   | 沙箱管理 | 单例模式，负责沙箱的创建、路由、生命周期管理                                                        |
| **Launcher 层**  | 沙箱启动 | 负责沙箱的启动、暂停、恢复、删除，如 `PreDeployLauncher`                                        |
| **Provider 层**  | 能力适配 | 实现 Protocol 接口，适配不同沙箱的 API，如 `AIOProvider`                                    |

**无状态设计**：Operation 是无状态的访问入口，沙箱是有状态的执行环境。沙箱内的文件、工作结果、环境变化等状态由沙箱自身维护，Operation 不负责持久化。

**Operation 与 Protocol**：Operation 面向用户，Protocol 面向实现者。Protocol 定义最小接口集，Provider 实现后由 Operation 组合提供上层能力。当前版本定义了 `BaseFsProtocol`、`BaseShellProtocol`、`BaseCodeProtocol` 三类。

## 2. 隔离粒度与生命周期

沙箱框架通过 `isolation_key` 决定请求命中哪个沙箱实例。隔离粒度决定沙箱的复用边界：

- 多个 Operation 实例，只要 `isolation_key` 相同，就会访问同一个沙箱
- 同一个 Operation 实例，在不同 session 上下文下，也可能访问不同沙箱

### 2.1 isolation\_key 格式

`isolation_key` 格式为：

```
{container_scope}_{launcher_type}_{sandbox_type}_{prefix}_{custom_id或session_id}
```

示例：

- `system_pre_deploy_aio___`：SYSTEM 级别
- `session_pre_deploy_aio_agent1__session123`：SESSION 级别，agent1 前缀
- `custom_pre_deploy_aio__my_sandbox_001`：CUSTOM 级别

Operation 在初始化时生成 `isolation_key` 模板，运行时由 Gateway Client 从协程上下文获取 `session_id` 并拼接成完整 `isolation_key`。

### 2.2 SYSTEM 粒度

所有请求共享同一个沙箱，`isolation_key` 固定，不包含动态部分。

**适用场景**：

- 单进程应用，全局共享一个沙箱实例
- 沙箱由外部服务管理，框架只负责连接

**生命周期**：

- 沙箱生命周期由外部管理
- 框架不会主动创建或销毁

### 2.3 SESSION 粒度

同一 `session_id` 共享同一个沙箱，不同 `session_id` 使用不同沙箱。`session_id` 从协程上下文动态获取，支持运行时动态创建沙箱。

**适用场景**：

- 多会话场景，每个会话独立隔离
- 同一会话内多个 agent 通过 `prefix` 进一步隔离

**配置示例**：

```python
# 同一会话内，agent1 和 agent2 使用不同沙箱
SandboxIsolationConfig(
    container_scope=ContainerScope.SESSION,
    prefix="agent1_",
)

SandboxIsolationConfig(
    container_scope=ContainerScope.SESSION,
    prefix="agent2_",
)
```

**生命周期**：

- 沙箱随会话动态创建
- 需要手动释放沙箱资源
- 支持配置 `on_stop` 策略：`delete`、`pause`、`keep`

### 2.4 CUSTOM 粒度

通过 `custom_id` 直接指定沙箱的逻辑身份，用户完全控制沙箱的复用边界。

**适用场景**：

- 需要固定命中某个沙箱
- 将沙箱生命周期绑定到业务 ID

**生命周期**：

- `custom_id` 完全由用户控制
- 需要手动释放沙箱资源

### 2.5 生命周期管理

SESSION 和 CUSTOM 粒度的沙箱需要手动释放资源。

**释放沙箱资源**：

```python
from openjiuwen.core.sys_operation.sandbox.gateway.gateway_client import SandboxGatewayClient

# 通过 isolation_key 释放
isolation_key = sys_op.isolation_key_template  # 从 SysOperation 获取
await SandboxGatewayClient.release(isolation_key, on_stop="delete")
```

**注意**：`isolation_key_template` 是模板，格式如 `session_pre_deploy_aio_agent1__{session_id}`。在 invoke 时，`{session_id}` 会从协程上下文自动填充；但在 release 时需要手动提供完整的 isolation_key。

> **说明**：当前版本已经具备 SESSION 粒度语义，但 gateway 的 session 持久化还没有实现"按 session 复用。

## 3. 配置详解

### 3.1 配置层级

`SandboxGatewayConfig` 是沙箱模式的顶层配置，包含隔离配置和 Launcher 配置：

```
SandboxGatewayConfig（顶层配置）
├── isolation: SandboxIsolationConfig（隔离配置）
│   ├── container_scope: ContainerScope（隔离级别）
│   ├── prefix: Optional[str]（命名空间前缀）
│   └── custom_id: Optional[str]（自定义容器 ID）
├── launcher_config: SandboxLauncherConfig（Launcher 配置）
│   ├── launcher_type: str（Launcher 类型）
│   ├── sandbox_type: str（沙箱类型）
│   ├── on_stop: Literal["delete", "pause", "keep"]（停止策略）
│   └── idle_ttl_seconds: Optional[int]（空闲超时）
└── timeout_seconds: int（超时设置）
```

### 3.2 隔离配置

`SandboxIsolationConfig` 决定请求的隔离与复用边界：

| 字段                | 类型             | 说明                             |
| ----------------- | -------------- | ------------------------------ |
| `container_scope` | ContainerScope | 隔离级别：SYSTEM / SESSION / CUSTOM |
| `prefix`          | Optional\[str] | 命名空间前缀，用于同一 scope 下进一步隔离       |
| `custom_id`       | Optional\[str] | 自定义容器 ID，仅在 CUSTOM 模式下使用       |

### 3.3 Launcher 配置

`SandboxLauncherConfig` 是 Launcher 配置基类，支持 `delete`、`pause`、`keep` 三种停止策略。

当前已实现的子类：

```
SandboxLauncherConfig（基类）
└── PreDeployLauncherConfig（预部署 Launcher 配置，用于连接已启动的沙箱服务）
    ├── launcher_type: Literal["pre_deploy"] = "pre_deploy"
    ├── sandbox_type: str（默认 "aio"）
    ├── base_url: str（沙箱服务地址）
    └── idle_ttl_seconds: Optional[int]（空闲超时）
```

## 4. 使用直连 AIO 沙箱

当前版本开箱即用的主路径是连接已启动的 AIO sandbox。AIO 是当前已接通的一种实现，不是框架绑定对象。

### 4.1 安装依赖

AIO 直连依赖 `agent-sandbox` 客户端，这部分在当前项目中是 optional extra。

从 PyPI 安装：

```bash
pip install -U "openjiuwen[sandbox]"
```

如果你是在源码仓中开发：

```bash
uv sync --extra sandbox
```

### 4.2 准备 AIO Sandbox 服务

运行 AIO 直连模式前，需要先准备一个可访问的 AIO sandbox 服务。

**使用 Docker 运行（推荐）**：

```bash
# 拉取并运行最新版本
docker run --security-opt seccomp=unconfined --rm -it -p 8080:8080 ghcr.io/agent-infra/sandbox:latest
```

> **说明**：更多 AIO Sandbox 的使用说明，请参考 [AIO Sandbox 快速开始](https://sandbox.agent-infra.com/zh/guide/start/quick-start)。

### 4.3 基本使用

连接已启动的 AIO 沙箱服务（SYSTEM 级别）：

```python
import asyncio
from openjiuwen.core.runner import Runner
from openjiuwen.core.sys_operation import (
    SysOperationCard,
    OperationMode,
    SandboxGatewayConfig,
    SandboxIsolationConfig,
    ContainerScope,
)
from openjiuwen.core.sys_operation.config import PreDeployLauncherConfig


async def main():
    # 1. 启动 Runner
    await Runner.start()

    try:
        # 2. 创建沙箱配置
        card = SysOperationCard(
            id="my_sandbox",
            mode=OperationMode.SANDBOX,
            gateway_config=SandboxGatewayConfig(
                isolation=SandboxIsolationConfig(
                    container_scope=ContainerScope.SYSTEM
                ),
                launcher_config=PreDeployLauncherConfig(
                    base_url="http://localhost:8080",
                    sandbox_type="aio",
                    idle_ttl_seconds=600,
                ),
                timeout_seconds=30,
            ),
        )

        # 3. 注册到资源管理器
        result = Runner.resource_mgr.add_sys_operation(card)
        assert result.is_ok()

        # 4. 获取 SysOperation 实例并执行操作
        sys_op = Runner.resource_mgr.get_sys_operation("my_sandbox")

        # 执行 Shell 命令
        shell_res = await sys_op.shell().execute_cmd(command="echo hello world")
        print(f"Shell 输出: {shell_res.data.stdout.strip()}")

        # 读取文件
        fs_res = await sys_op.fs().read_file(path="/etc/hosts")
        print(f"文件内容长度: {len(fs_res.data.content)}")

        # 执行 Python 代码
        code_res = await sys_op.code().execute_code(
            code="print('Hello from Python')",
            language="python"
        )
        print(f"代码输出: {code_res.data.stdout.strip()}")

    finally:
        # 5. 清理资源
        Runner.resource_mgr.remove_sys_operation(sys_operation_id="my_sandbox")
        await Runner.stop()


if __name__ == "__main__":
    asyncio.run(main())
```

**关键配置说明**：

- `mode=OperationMode.SANDBOX`：指定运行模式为沙箱模式
- `SandboxIsolationConfig`：决定请求的隔离与复用边界
- `PreDeployLauncherConfig`：表示连接一个已经存在的 sandbox 服务
- `sandbox_type="aio"`：表示由 AIO provider 负责把统一接口映射到 AIO 沙箱

### 4.4 作为Agent工具调用

注册沙箱后，其方法会自动包装为 `LocalFunction` 工具，可通过 Tool ID 获取并添加到 Agent：

```python
# 获取工具实例
tool_id = SysOperationCard.generate_tool_id("my_sandbox", "shell", "execute_cmd")
execute_cmd_tool = Runner.resource_mgr.get_tool(tool_id)

# 调用工具
res = await execute_cmd_tool.invoke({"command": "echo hello"})
print(res.data.stdout.strip())

# 添加到 Agent
agent.ability_manager.add(execute_cmd_tool.card)
```

## 5. 扩展自定义沙箱

沙箱框架的扩展点分为两类：

| 扩展点          | 职责             |
| ------------ | -------------- |
| **Launcher** | 沙箱实例的获取与生命周期管理 |
| **Provider** | 沙箱能力的协议实现与适配   |

两者正交组合：

- 如果沙箱获取方式不变，只需扩展 Provider
- 如果沙箱获取方式也需要自定义，则需同时扩展 Launcher 和 Provider

### 5.1 扩展 Launcher

Launcher 负责沙箱的启动、暂停、恢复和删除，对应的核心基类是 `SandboxLauncher`。

```python
from openjiuwen.core.sys_operation.sandbox.launchers.base import SandboxLauncher, LaunchedSandbox
from openjiuwen.core.sys_operation.config import SandboxLauncherConfig


class MyCustomLauncher(SandboxLauncher):
    async def launch(
        self,
        config: SandboxLauncherConfig,
        timeout_seconds: int,
        isolation_key: str | None = None,
    ) -> LaunchedSandbox:
        # 实现沙箱启动逻辑
        # 返回 LaunchedSandbox(base_url="...", sandbox_id="...")
        pass

    async def pause(self, sandbox_id: str) -> None:
        # 实现沙箱暂停逻辑
        pass

    async def resume(self, sandbox_id: str) -> None:
        # 实现沙箱恢复逻辑
        pass

    async def delete(self, sandbox_id: str) -> None:
        # 实现沙箱删除逻辑
        pass
```

### 5.2 扩展 Provider

Provider 需要实现 Protocol 定义的接口。Protocol 定义了沙箱底层需要实现的最小接口集。

**Protocol 类型**：

| Protocol            | 说明       | 基础方法                                    |
| ------------------- | -------- | --------------------------------------- |
| `BaseFsProtocol`    | 文件系统协议   | `read_file`、`write_file`、`list_files` 等 |
| `BaseShellProtocol` | Shell 协议 | `execute_cmd`、`execute_cmd_stream` 等    |
| `BaseCodeProtocol`  | 代码执行协议   | `execute_code`、`execute_code_stream` 等  |

**扩展要点**：

- Provider 只需实现 Protocol 定义的基础方法
- Operation 层的高级功能基于这些基础方法实现
- 不必一次性实现所有能力
- 理论上，只实现 Shell Protocol，也可以基于 Shell 组合出其它上层行为

Provider 负责实现 Protocol 接口，适配具体沙箱的 API：

```python
from openjiuwen.core.sys_operation.sandbox.providers.base_provider import BaseFSProvider
from openjiuwen.core.sys_operation.sandbox.gateway.gateway import SandboxEndpoint
from openjiuwen.core.sys_operation.result import ReadFileResult


class MyCustomFSProvider(BaseFSProvider):
    """实现 BaseFsProtocol 接口的文件系统 Provider"""
    
    def __init__(self, endpoint: SandboxEndpoint, config=None):
        super().__init__(endpoint, config)
        # 初始化沙箱客户端

    async def read_file(
        self,
        path: str,
        mode: str = "text",
        **kwargs
    ) -> ReadFileResult:
        # 实现文件读取逻辑（Protocol 要求的基础方法）
        # 调用沙箱 API
        pass

    # 实现其他 Protocol 方法...
```

### 5.3 完整扩展示例

这个扩展示例体现了完整链路：

1. 定义自定义 `SandboxLauncherConfig`
2. 定义 `SandboxLauncher`
3. 定义 `Provider`
4. 注册 Launcher 和 Provider
5. 在 `SysOperationCard` 中引用新的 `launcher_type` 和 `sandbox_type`
6. 最终仍通过统一的 `sys_op.shell()` 接口调用

如果需要支持文件系统或代码执行，补充对应的 `fs`、`code` provider 实现即可。

以下示例只实现 `shell` 能力，说明可以从最小能力集开始扩展：

```python
import asyncio
from typing import Any, AsyncIterator, Dict, Literal, Optional

from pydantic import Field

from openjiuwen.core.common.exception.codes import StatusCode
from openjiuwen.core.runner import Runner
from openjiuwen.core.sys_operation import OperationMode, SandboxGatewayConfig, SysOperationCard
from openjiuwen.core.sys_operation.config import (
    ContainerScope,
    SandboxIsolationConfig,
    SandboxLauncherConfig,
)
from openjiuwen.core.sys_operation.result import (
    ExecuteCmdChunkData,
    ExecuteCmdData,
    ExecuteCmdResult,
    ExecuteCmdStreamResult,
)
from openjiuwen.core.sys_operation.sandbox.launchers.base import LaunchedSandbox, SandboxLauncher
from openjiuwen.core.sys_operation.sandbox.providers.base_provider import BaseShellProvider
from openjiuwen.core.sys_operation.sandbox.sandbox_registry import SandboxRegistry


# 步骤 1: 定义自定义配置
class MySandboxLauncherConfig(SandboxLauncherConfig):
    launcher_type: Literal["my_launcher"] = "my_launcher"
    sandbox_type: str = Field(default="my_sandbox")
    base_url: str = Field(..., description="自定义 sandbox 服务地址")


# 步骤 2: 定义 Launcher
class MySandboxLauncher(SandboxLauncher):
    async def launch(
        self,
        config: SandboxLauncherConfig,
        timeout_seconds: int,
        isolation_key: Optional[str] = None,
    ) -> LaunchedSandbox:
        if not isinstance(config, MySandboxLauncherConfig):
            raise ValueError("MySandboxLauncher requires MySandboxLauncherConfig")
        return LaunchedSandbox(
            base_url=config.base_url,
            sandbox_id=isolation_key,
        )


# 步骤 3: 定义 Provider
class MySandboxShellProvider(BaseShellProvider):
    async def execute_cmd(
        self,
        command: str,
        *,
        cwd: Optional[str] = None,
        timeout: Optional[int] = 300,
        environment: Optional[Dict[str, str]] = None,
        options: Optional[Dict[str, Any]] = None,
    ) -> ExecuteCmdResult:
        return ExecuteCmdResult(
            code=StatusCode.SUCCESS.code,
            message=StatusCode.SUCCESS.errmsg,
            data=ExecuteCmdData(
                command=command,
                cwd=cwd or ".",
                exit_code=0,
                stdout=f"[my_sandbox] {command}\n",
                stderr="",
            ),
        )

    async def execute_cmd_stream(
        self,
        command: str,
        *,
        cwd: Optional[str] = None,
        timeout: Optional[int] = 300,
        environment: Optional[Dict[str, str]] = None,
        options: Optional[Dict[str, Any]] = None,
    ) -> AsyncIterator[ExecuteCmdStreamResult]:
        yield ExecuteCmdStreamResult(
            code=StatusCode.SUCCESS.code,
            message=StatusCode.SUCCESS.errmsg,
            data=ExecuteCmdChunkData(
                text=f"[my_sandbox] {command}\n",
                type="stdout",
                chunk_index=0,
                exit_code=0,
            ),
        )


# 步骤 4: 注册 Launcher 和 Provider
SandboxRegistry.register_launcher("my_launcher", MySandboxLauncher)
SandboxRegistry.register_provider("my_sandbox", "shell", MySandboxShellProvider)


# 步骤 5: 在配置中引用自定义组件
async def main():
    await Runner.start()

    card_id = "my_sandbox_op"

    try:
        card = SysOperationCard(
            id=card_id,
            mode=OperationMode.SANDBOX,
            gateway_config=SandboxGatewayConfig(
                isolation=SandboxIsolationConfig(
                    container_scope=ContainerScope.CUSTOM,
                    custom_id="demo_shell_sandbox",
                ),
                launcher_config=MySandboxLauncherConfig(
                    base_url="http://localhost:9000",
                    idle_ttl_seconds=600,
                ),
                timeout_seconds=30,
            ),
        )

        add_res = Runner.resource_mgr.add_sys_operation(card)
        assert add_res.is_ok()

        sys_op = Runner.resource_mgr.get_sys_operation(card_id)
        res = await sys_op.shell().execute_cmd(command="echo hello")

        if res.code == 0:
            print(res.data.stdout.strip())
    finally:
        Runner.resource_mgr.remove_sys_operation(sys_operation_id=card_id)
        await Runner.stop()


asyncio.run(main())
```
