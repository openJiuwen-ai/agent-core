# 异步回调框架

异步回调框架（Async Callback Framework）是 openJiuwen 中面向 asyncio 的生产级事件驱动框架，用于在 Runner 或业务中注册事件回调、按优先级执行、做限流/熔断/校验等过滤、以及链式执行与回滚。Runner 内置了该框架，可通过 `Runner.callback_framework` 访问。

**API 文档**：完整类与方法说明见 [callback API 文档](../API文档/openjiuwen.core/runner/callback/callback.README.md)。

---

## 概述

### 核心特性

- **异步优先**：完全基于 asyncio，支持高并发。
- **优先级控制**：数字越大越先执行，便于安排关键步骤与辅助步骤。
- **过滤系统**：全局 / 事件 / 回调三级过滤器，8 种内置过滤器，支持自定义。
- **回调链与回滚**：多步顺序执行，上一步结果传给下一步；任一步失败可触发已执行步骤的回滚。
- **多种触发模式**：顺序、并行、条件、超时、延迟、流式、生成器聚合共 7 种。
- **装饰器**：`@on` 注册，以及 `trigger_on_call`、`emits`、`emit_around`、`emits_stream`、`transform_io` 等自动触发与 I/O 变换。
- **性能监控**：调用次数、耗时、错误率等指标，慢回调排查。
- **生命周期钩子**：BEFORE（执行前）、AFTER（执行后）、ERROR（出错时）、CLEANUP（清理时）。
- **异步生成器**：支持流式输入（`trigger_stream`）与流式/多源聚合输出（`trigger_generator`、`emits_stream`）。
- **输入输出变换**：通过事件或直接回调对函数入参和返回值做变换，支持普通函数与生成器。

### 模块结构

框架代码位于 `openjiuwen/core/runner/callback/`：

- `framework.py`：核心类 `AsyncCallbackFramework`
- `enums.py`：`FilterAction`、`ChainAction`、`HookType`
- `models.py`：`CallbackMetrics`、`FilterResult`、`ChainContext`、`ChainResult`、`CallbackInfo`
- `filters.py`：内置过滤器实现
- `chain.py`：回调链与回滚
- `decorator.py`：各类装饰器实现

## 快速开始

通过 Runner 使用回调框架：

```python
import asyncio
from openjiuwen.core.runner import Runner

framework = Runner.callback_framework

@framework.on("user_login", priority=10)
async def send_welcome(username: str):
    print(f"Sending welcome to {username}")
    return f"welcome_{username}"

async def main():
    results = await framework.trigger("user_login", username="alice")
    print(results)  # ['welcome_alice']

asyncio.run(main())
```

不依赖 Runner 时，可单独创建框架实例：

```python
from openjiuwen.core.runner.callback import AsyncCallbackFramework

framework = AsyncCallbackFramework(enable_metrics=True)
@framework.on("my_event", priority=5)
async def handler(x: int):
    return x * 2

results = await framework.trigger("my_event", 3)  # [6]
```

## 动态注册回调（非装饰器方式）

除装饰器 `@framework.on(...)` 外，可在运行时通过 **register** 动态注册回调，适用于根据配置、插件或条件在运行期增减回调的场景。

### 何时使用

- **运行时按条件注册**：根据配置开关、环境变量或用户权限决定是否注册某回调。
- **插件/扩展**：插件加载时把自身提供的异步函数注册到框架，无需在定义处使用装饰器。
- **配置驱动**：从配置文件或数据库读取“事件名 + 回调引用/路径”，在启动或重载时批量 `register`。
- **临时注册**：在单次请求或会话内注册，用完后通过 **unregister** 移除，避免长期占用。

### register 用法

`register` 为异步方法，参数与 `@framework.on()` 一致：

```python
await framework.register(
    "event_name",           # 事件名
    async_callback_func,    # 异步回调函数（可调用对象）
    priority=0,
    once=False,
    namespace="default",
    tags=None,              # 如 {"tag1", "tag2"}
    filters=None,           # 如 [RateLimitFilter(...)]
    rollback_handler=None,
    error_handler=None,
    max_retries=0,
    retry_delay=0.0,
    timeout=None,
)
```

示例：根据配置动态注册多个回调

```python
import asyncio
from openjiuwen.core.runner import Runner

framework = Runner.callback_framework

async def notify_email(user: str, **kwargs):
    print(f"[Email] Notify {user}")
    return "email_ok"

async def notify_sms(user: str, **kwargs):
    print(f"[SMS] Notify {user}")
    return "sms_ok"

# 从配置或数据库读取：哪些渠道在运行时启用
enabled_channels = ["email", "sms"]  # 示例：仅启用 email、sms

channel_handlers = {
    "email": (notify_email, 20),
    "sms": (notify_sms, 10),
}

async def setup_callbacks():
    for ch in enabled_channels:
        if ch in channel_handlers:
            handler, priority = channel_handlers[ch]
            await framework.register("user_signup", handler, priority=priority, namespace="channels")
    print("Callbacks registered.")

async def main():
    await setup_callbacks()
    results = await framework.trigger("user_signup", user="alice")
    print(results)  # 例如 ['email_ok', 'sms_ok']

asyncio.run(main())
```

### 注销方式

- **unregister(event, callback)**：从指定事件上移除一个回调（可传入原始函数或 `@on` 返回的包装函数）。
- **unregister_namespace(namespace)**：移除该命名空间下所有回调。
- **unregister_by_tags(tags)**：移除带有给定标签中任意一个的回调。
- **unregister_event(event)**：移除该事件下的全部回调，并清理该事件关联的过滤器、链、钩子与熔断器。

动态注册后若需在适当时机收回，可保存回调引用并在用完后调用 `unregister`：

```python
await framework.register("temp_event", my_async_handler, namespace="temp")
# ... 使用 ...
await framework.unregister("temp_event", my_async_handler)

# 或按命名空间整批收回
await framework.unregister_namespace("temp")
```

## 核心概念

### 事件与回调

- **事件**：字符串标识（如 `"user_login"`、`"order.created"`），用于绑定一组回调。
- **回调**：异步函数，在事件触发时按优先级执行。同一事件可注册多个回调。
- **优先级**：`@framework.on(event, priority=N)` 中 N 越大越先执行；未指定时默认为 0。

建议事件命名使用有意义的点分形式（如 `user.login`、`order.created`），避免 `event1`、`do_something` 等泛用名。

### 执行流程（顺序触发）

1. 检查事件是否已注册；未注册则返回空列表。
2. 按优先级排序该事件下的回调。
3. 执行 **BEFORE** 钩子。
4. 遍历回调：若未启用则跳过；否则依次经过**全局过滤器 → 事件过滤器 → 该回调专属过滤器**。
5. 过滤器可返回：**CONTINUE**（继续）、**STOP**（停止整个事件）、**SKIP**（跳过当前回调）、**MODIFY**（用新参数继续）。
6. 执行回调，收集结果；若设置了 `once=True` 则执行后禁用该回调。
7. 若某回调抛错，执行 **ERROR** 钩子，并继续处理后续回调（不中断整次触发）。
8. 全部完成后执行 **AFTER** 钩子；若有 CLEANUP 钩子则在 finally 中执行。

### 过滤器层级

- **全局过滤器**：`add_global_filter(filter)`，对所有事件生效，适合日志、监控。
- **事件过滤器**：`add_filter(event, filter)`，仅对指定事件生效，适合限流、鉴权。
- **回调过滤器**：在 `@framework.on(event, filters=[...])` 中传入，仅对该回调生效。

执行顺序为：全局 → 事件 → 回调；任一层返回 STOP 或 SKIP 会短路后续执行或跳过当前回调。

### 链式执行与回滚

- 使用 **trigger_chain** 时，回调按优先级顺序执行，**上一个回调的返回值**作为下一个回调的**第一个位置参数**，其余
  `initial_kwargs` 不变；链中可通过 `kwargs["_chain_context"]` 访问 `ChainContext`（如 `get_last_result()`、
  `set_metadata()`）。
- 每个回调可返回 **ChainResult(ChainAction.CONTINUE, result=...)** 表示继续，或 **ChainResult(ChainAction.ROLLBACK,
  error=...)** 表示回滚。
- **ChainAction**：CONTINUE（继续下一回调）、BREAK（中断链并返回当前结果）、RETRY（重试当前回调）、ROLLBACK（回滚）。
- 一旦返回 ROLLBACK 或回调抛错，框架会**按执行顺序的逆序**调用已执行回调注册的 **rollback_handler**，用于释放资源、退款、恢复库存等。

## 触发模式

| 方法                       | 说明                                           | 适用场景          |
|--------------------------|----------------------------------------------|---------------|
| **trigger**              | 按优先级顺序执行，收集所有结果                              | 默认方式，有依赖或需顺序时 |
| **trigger_parallel**     | 并发执行，I/O 密集时可显著缩短总耗时                         | 多路拉取、并行请求     |
| **trigger_until**        | 按顺序执行，直到某回调返回值使 condition(result) 为 True 即返回 | 降级、首选/备选      |
| **trigger_chain**        | 链式执行，上一步结果传下一步，支持回滚                          | 订单/支付等事务流程    |
| **trigger_with_timeout** | 在给定总时间内执行 trigger，超时返回已收集结果                  | 控制最大耗时、SLA    |
| **trigger_delayed**      | 延迟指定秒数后执行 trigger                            | 定时、延迟任务       |
| **trigger_stream**       | 对异步输入流逐项触发事件并流式产出结果                          | 大文件、日志流、传感器数据 |
| **trigger_generator**    | 执行事件下所有回调，聚合各回调的异步生成器或普通返回值并逐项 yield         | 多源数据聚合、流式 API |

**性能建议**：I/O 密集型（如多路请求、数据库查询）优先用 **trigger_parallel**；CPU 密集型或存在共享状态时用 **trigger**
。典型场景下并行可比顺序快约 2～4 倍（例如 3 个 0.15s 的 I/O 回调：顺序约 0.45s，并行约 0.2s）。

## 流式与生成器

### trigger_stream：流式输入

对**异步输入流**中的每一项触发一次事件，逐项执行回调并流式产出结果，无需将全部数据加载到内存。

- 适用：大文件逐行/逐块处理、实时日志分析、流式数据管道。
- 回调可返回普通值或异步生成器；若返回生成器，其 yield 出的每一项都会作为该次触发的产出。
- 单个 item 处理失败不会中断整条流，错误会被记录到日志和指标，失败的 item 不产生输出。

示例：流式处理日志

```python
async def log_stream():
    with open("app.log") as f:
        for line in f:
            yield line.strip()

@framework.on("process_log")
async def analyze_log(line: str):
    if "ERROR" in line:
        return {"level": "error", "line": line}
    return {"level": "info", "line": line}

async for analysis in framework.trigger_stream("process_log", log_stream()):
    if analysis["level"] == "error":
        await alert_admin(analysis)
```

### trigger_generator：生成器输出聚合

执行某事件下所有回调，若回调返回**异步生成器**则迭代其产出，否则将返回值作为单项产出；按优先级顺序逐项 yield，实现多源聚合。

- 适用：多源数据聚合（DB + 缓存 + API）、流式 API 响应、报表分段生成。

示例：多源用户数据聚合

```python
@framework.on("fetch_users")
async def from_database():
    users = await db.query("SELECT * FROM users")
    for user in users:
        yield user

@framework.on("fetch_users")
async def from_cache():
    for user in await redis.get_all("user:*"):
        yield user

async for user in framework.trigger_generator("fetch_users"):
    process(user)
```

### emits_stream：装饰器流式触发

装饰**异步生成器函数**，在每次 **yield** 时自动触发指定事件，并将 yield 出的项以关键字参数（默认键名 `item`
）传给该事件的回调；原生成器仍向调用方正常 yield。

- 适用：实时指标发布、日志流监控、进度通知、事件溯源。

流式相比批处理的特点：内存占用 O(1)、首字节延迟低、适合无限长数据、实时性好。

## 过滤器

### 内置过滤器一览

| 过滤器                      | 功能                                                                           | 典型用途       |
|--------------------------|------------------------------------------------------------------------------|------------|
| **RateLimitFilter**      | 在时间窗口内限制调用次数                                                                 | 防止回调执行过于频繁 |
| **CircuitBreakerFilter** | 失败次数达到阈值后熔断，超时后尝试恢复                                                          | 保护下游服务     |
| **ValidationFilter**     | 用自定义函数校验参数                                                                   | 参数合法性检查    |
| **LoggingFilter**        | 记录事件、回调名、参数                                                                  | 审计、调试      |
| **AuthFilter**           | 要求 kwargs 中 `user_role` 等于指定角色                                               | 基于角色的访问控制  |
| **ParamModifyFilter**    | 用 modifier(*args, **kwargs) 返回 (new_args, new_kwargs) 修改参数                   | 转换或增强参数    |
| **ConditionalFilter**    | 根据 condition(event, callback, *args, **kwargs) 决定 CONTINUE 或 action_on_false | 条件执行       |

均来自 `openjiuwen.core.runner.callback`，继承自 **EventFilter**；自定义过滤器可子类化 `EventFilter` 并实现
`async def filter(self, event, callback, *args, **kwargs) -> FilterResult`。

### 示例：限流与鉴权

```python
from openjiuwen.core.runner import Runner
from openjiuwen.core.runner.callback import RateLimitFilter, AuthFilter

framework = Runner.callback_framework
framework.add_filter("api_call", RateLimitFilter(max_calls=100, time_window=60.0))
framework.add_filter("admin_action", AuthFilter(required_role="admin"))

@framework.on("admin_action")
async def do_admin(user_id: str, user_role: str):
    return f"admin_ok_{user_id}"

await framework.trigger("admin_action", user_id="u1", user_role="admin")
```

### 熔断器

使用 **add_circuit_breaker(event, callback, failure_threshold=5, timeout=60.0)** 可为某事件的指定回调绑定熔断器；框架在回调成功/失败时自动调用
`record_success` / `record_failure`。失败次数达到阈值后该回调会被跳过，直到超过 timeout 后再次尝试。

## 回调链与回滚

用 **trigger_chain** 和 **ChainResult** / **ChainAction** 实现多步流程；任一步返回 **ChainAction.ROLLBACK** 或抛错会按*
*执行顺序的逆序**执行已注册的 **rollback_handler**（如退款、恢复库存）。

```python
from openjiuwen.core.runner import Runner
from openjiuwen.core.runner.callback import ChainAction, ChainResult

framework = Runner.callback_framework

async def rollback_payment(ctx):
    print("Rollback payment")

async def rollback_inventory(ctx):
    print("Rollback inventory")

@framework.on("order_create", priority=30, rollback_handler=rollback_payment)
async def pay(order: dict, **kwargs):
    order["paid"] = True
    return ChainResult(ChainAction.CONTINUE, result=order)

@framework.on("order_create", priority=20, rollback_handler=rollback_inventory)
async def reserve(order: dict, **kwargs):
    if order.get("quantity", 0) > 100:
        return ChainResult(ChainAction.ROLLBACK, error=Exception("Insufficient stock"))
    order["reserved"] = True
    return ChainResult(ChainAction.CONTINUE, result=order)

result = await framework.trigger_chain("order_create", order={"quantity": 150})
if result.action == ChainAction.ROLLBACK:
    print("Order failed, rolled back")
```

链中可通过 `kwargs["_chain_context"]` 获取 **ChainContext**，使用 `get_last_result()`、`set_metadata(key, value)`、
`get_metadata(key)` 等在步骤间传递数据。

## 装饰器

### 注册与自动触发

- **@framework.on(event, priority=0, once=False, namespace="default", tags=None, filters=None, rollback_handler=None,
  error_handler=None, max_retries=0, retry_delay=0.0, timeout=None, callback_type="")**
  将函数注册为事件的回调。priority 越大越先执行；once 为 True 时执行一次后自动禁用；error_handler 在该回调抛错时被调用；callback_type 为语义类型标记，`"transform"` 表示 transform 专用回调。

- **@framework.trigger_on_call(event, pass_result=False, pass_args=True)**
  被装饰函数**被调用时**先触发事件（可选传入参数或返回后再触发一次并传 result）。

- **@framework.emits(event, result_key="result", include_args=False)**
  函数**返回后**触发事件，以 result_key 为键传入返回值；可选是否带上原始参数。

- **@framework.emit_around(before_event, after_event, pass_args=True, pass_result=True, on_error_event=None)**
  执行前触发 before_event，执行成功后触发 after_event；若抛错且指定了 on_error_event 则触发该事件。

- **@framework.emits_stream(event, item_key="item")**
  装饰**异步生成器**，每次 yield 时触发 event，并以 item_key 为键传入该项。

- **@framework.on_transform(event, *, priority=0)**
  注册 transform 类型回调的便捷装饰器，等价于 `@framework.on(event, callback_type="transform")`。仅被 `trigger_transform` 和 `transform_io` 事件模式触发；普通 `trigger` 不会调用此类回调。

### transform_io：输入输出变换

对函数的**入参**和**返回值**做统一变换，支持两种方式：

1. **事件模式**：设置 `input_event` / `output_event`。调用被装饰函数前先触发 input_event，回调接收 `(*args, **kwargs)` 并返回
   `(new_args, new_kwargs)`，框架用最后一个回调结果作为新参数；返回后（或生成器每项）触发 output_event，回调接收
   `result_key=value` 并返回新值，框架用最后一个回调结果作为最终结果。**注意**：事件模式中只触发 `callback_type="transform"` 的回调，必须用 `@framework.on_transform(event)` 或 `@framework.on(event, callback_type="transform")` 注册；普通 `@framework.on` 注册的回调不会被 `transform_io` 触发。
2. **直接回调模式**：设置 `input_transform` / `output_transform`。`input_transform(*args, **kwargs)` 返回
   `(new_args, new_kwargs)`；`output_transform(value)` 返回新值。支持同步/异步。

`output_mode` 控制对生成器函数的输出变换方式：

* **`'frame'`**（默认）：**逐帧模式**，对每个 yield 项单独调用 output 变换；事件模式下 output_event 对每项触发一次。
* **`'generator'`**：**生成器模式**，将整个 source 异步迭代器一次性传给 output 变换。直接回调模式下 `output_transform` 须为异步生成器函数 `(source: AsyncIterator) -> AsyncIterator`，可对每项 yield 0/1/多次，支持过滤、展开、有状态变换；事件模式下 output_event 触发一次，回调接收 source 并返回可迭代对象；同步生成器自动提升为异步。对非生成器函数使用时抛出 `ValueError`。

适用于参数规范化、结果序列化、统一加解密等。示例（事件模式）：

```python
@framework.on_transform("transform_input")
async def normalize_input(*args, **kwargs):
    return (args, {**kwargs, "limit": kwargs.get("limit", 10)})

@framework.on_transform("transform_output")
async def serialize_output(result):
    return json.dumps(result) if isinstance(result, dict) else result

@framework.transform_io(
    input_event="transform_input",
    output_event="transform_output",
)
async def fetch_data(limit: int):
    return {"count": limit}
```

示例（生成器模式 — 直接回调）：

```python
async def expand_chunks(source):
    """将每个分块拆分为独立 token。"""
    async for chunk in source:
        for token in chunk.split():
            yield token

@framework.transform_io(output_transform=expand_chunks, output_mode="generator")
async def stream_response():
    yield "hello world"
    yield "foo bar baz"

# 实际 yield: "hello", "world", "foo", "bar", "baz"
```

## 钩子与指标

### 生命周期钩子

- **HookType.BEFORE**：事件下所有回调执行前。
- **HookType.AFTER**：所有回调执行完毕后（传入结果列表等）。
- **HookType.ERROR**：某个回调抛错时（传入异常等）。
- **HookType.CLEANUP**：在 finally 中执行，用于收尾。

通过 **add_hook(event, hook_type, hook)** 注册；hook 可为同步或异步函数。

### 指标与慢回调

- **get_metrics(event=None, callback=None)**：返回各回调的
  call_count、avg_time、min_time、max_time、error_count、error_rate、last_call_time 等。
- **reset_metrics()**：清空所有指标。
- **get_slow_callbacks(threshold=1.0)**：返回平均耗时超过阈值的回调列表，便于性能排查。

建议在关键事件上开启 `enable_metrics=True`（默认开启），并定期查看慢回调。

## Tool / LLM / Workflow 的 invoke/stream 拦截

`Tool`、`BaseModelClient`、`Workflow` 在实例化时自动将 `invoke` 和 `stream` 方法包装进对应的 `transform_io` 管道。注册相应事件的 transform 回调即可全局拦截所有实例的调用。

### 可用事件

| 类 | 输入事件 | 输出事件 |
|----|----------|----------|
| `Tool.invoke` | `ToolCallEvents.TOOL_INVOKE_INPUT` | `ToolCallEvents.TOOL_INVOKE_OUTPUT` |
| `Tool.stream` | `ToolCallEvents.TOOL_STREAM_INPUT` | `ToolCallEvents.TOOL_STREAM_OUTPUT` |
| `BaseModelClient.invoke` | `LLMCallEvents.LLM_INVOKE_INPUT` | `LLMCallEvents.LLM_INVOKE_OUTPUT` |
| `BaseModelClient.stream` | `LLMCallEvents.LLM_STREAM_INPUT` | `LLMCallEvents.LLM_STREAM_OUTPUT` |
| `Workflow.invoke` | `WorkflowEvents.WORKFLOW_INVOKE_INPUT` | `WorkflowEvents.WORKFLOW_INVOKE_OUTPUT` |
| `Workflow.stream` | `WorkflowEvents.WORKFLOW_STREAM_INPUT` | `WorkflowEvents.WORKFLOW_STREAM_OUTPUT` |

输入回调接收 `(*args, **kwargs)` 并返回 `(new_args, new_kwargs)`；输出回调接收 `result=<value>` 并返回新值；stream 的输出事件对每个 yield item 独立触发一次。

### 示例

```python
from openjiuwen.core.runner import Runner
from openjiuwen.core.runner.callback.events import ToolCallEvents

fw = Runner.callback_framework

# 拦截所有 Tool.invoke 的入参
@fw.on_transform(ToolCallEvents.TOOL_INVOKE_INPUT)
async def preprocess(inputs, **kwargs):
    # inputs 是 Tool.invoke 的第一个位置参数
    return ((inputs,), kwargs)

# 拦截所有 Tool.invoke 的返回值
@fw.on_transform(ToolCallEvents.TOOL_INVOKE_OUTPUT)
async def postprocess(result):
    return result  # 原样透传，或修改后返回
```

回调在 **所有** Tool/LLM/Workflow 实例上生效，无需修改各实例代码。

## 与 Runner 的配合

Runner 在启动后通过 **Runner.callback_framework** 暴露同一套框架实例。可在 Agent/Workflow 执行前后注册事件（如
`workflow_started`、`agent_finished`），用于日志、指标、审计或扩展业务流程。Runner 的配置仍通过 **Runner.set_config** / *
*Runner.get_config** 管理，与回调框架解耦。

## 最佳实践

1. **事件命名**：使用有语义的点分形式（如 `user.login`、`order.created`），避免泛用名。
2. **优先级**：关键步骤（如鉴权、校验）设高 priority，辅助步骤（如通知、统计）设低 priority。
3. **I/O 与 CPU**：I/O 密集用 **trigger_parallel**；CPU 密集或存在共享状态时用 **trigger**。
4. **命名空间与标签**：用 `namespace`、`tags` 组织回调，便于 **unregister_namespace**、**unregister_by_tags** 批量管理。
5. **错误处理**：对易失败的回调设置 **error_handler**，或在链中使用 **rollback_handler** 保证一致性。
6. **事务型流程**：多步且需“全成功或全回滚”时使用 **trigger_chain** 与 **rollback_handler**。
7. **监控**：定期用 **get_slow_callbacks(threshold)** 排查慢回调，必要时为回调设置 **timeout**。

## 常见问题

- **何时用 trigger 还是 trigger_parallel？**  
  有顺序依赖或共享状态时用 trigger；回调相互独立且 I/O 密集时用 trigger_parallel。

- **过滤器执行顺序？**  
  全局过滤器 → 事件过滤器 → 回调专属过滤器；任一层可返回 STOP（停止整次触发）或 SKIP（跳过当前回调）。

- **链中数据如何传递？**  
  下一个回调的第一个位置参数是上一个回调的返回值；此外可通过 `kwargs["_chain_context"]` 的 `set_metadata` /
  `get_metadata`、`get_last_result()` 传递更多状态。

- **如何实现“仅当条件满足才执行”？**  
  使用 **ConditionalFilter**，或使用 **trigger_until** 在找到第一个满足条件的结果后停止。

- **流式处理中单条失败会怎样？**  
  单条失败不会中断流；错误会记入日志和指标，该条不产生输出，后续 item 继续处理。
