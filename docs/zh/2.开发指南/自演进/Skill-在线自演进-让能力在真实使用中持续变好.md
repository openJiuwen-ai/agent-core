# Skill 在线自演进：让能力在真实使用中持续变好

Agent Skills 和 Swarm Skills 解决的是“能力如何沉淀”的问题：前者沉淀单个 Agent 的专业能力，后者沉淀多 Agent 团队的协作方式。

但只“沉淀”还不够。

真实世界里的任务会不断变化。工具接口会报错，文件格式会变，用户会纠正 Agent 的理解，团队协作中也会暴露新的边界情况。如果 Skill 创建完成后就不再变化，它迟早会变成一份过期说明书。

Skill 在线自演进解决的就是这个问题。

一句话理解：

> 创建 Skill，是让 Agent 获得一项能力；在线自演进，是让这项能力在真实使用中持续变好。

它不是让系统偷偷改文件，也不是让 Agent 无限制地重写自己。openJiuwen 的自演进是一套透明、可管理、关键变更经过确认、重要版本可回滚的经验闭环。

---

## 先划清边界：这里只讲已有能力的演进

这里讨论的是：**已经存在的 Agent Skill 或 Swarm Skill，如何在真实使用中持续变好**。

如果当前还没有合适的 Skill，系统根据任务轨迹建议创建一个新 Skill 或 Swarm Skill，那属于“自动建议创建”流程，目标是从无到有沉淀新能力。

在线自演进关注的是另一个问题：

> 这个已经存在的能力，在使用中暴露了问题或积累了经验，是否应该补充、修正、整理或重建？

---

## 自演进到底在演进什么？

很多人听到“自演进”，第一反应是：Agent 会不会自己乱改 `SKILL.md`？

openJiuwen 不是这样设计的。

它日常演进的核心对象是“经验记录”，而不是直接覆盖原始 Skill 的主体流程。

一个 Skill 的能力可以分成两层：

```text
Skill 原始定义
  └── SKILL.md、roles/、workflow.md、scripts/ 等基础文件

演进经验层
  └── evolutions.json 中的经验记录
```

原始定义回答：

> 这个 Skill 最初应该如何工作？

演进经验回答：

> 在真实使用中，我们又学到了哪些补充规则、错误处理、用户偏好和边界条件？

这种设计很关键。

如果每次演进都直接改 `SKILL.md`，会出现三个问题：

- 改错了不好回退。
- 新旧经验混在一起，无法独立评估每条经验是否有效。
- 基础 Skill 升级时，历史改动容易冲突。

所以 openJiuwen 采用的是“演进补丁架构”：

> 新经验先作为独立记录附加到 Skill 上，在运行时动态加载；系统可能更新 Skill 中的经验索引或展示块，但不会把每条新经验直接改写进主体流程。只有在用户明确发起重建时，才会把高质量经验重新组织进 Skill 主体。

---

## 演进补丁长什么样？

演进经验会存放在 Skill 目录下的 `evolutions.json` 中。

一个经验记录通常包含：

- `id`：经验 ID。
- `source`：来源，例如执行失败、用户纠错、主动演进。
- `timestamp`：生成时间。
- `context`：当时发生了什么。
- `change`：建议补充到哪里、补充什么内容。
- `score`：质量分数。
- `usage_stats`：后续使用统计。
- `applied`：是否已经固化进主 Skill。

示例：

```json
{
  "entries": [
    {
      "id": "ev_1cdbc3a5",
      "source": "execution_failure",
      "timestamp": "2026-03-09T09:33:08Z",
      "context": "导出 Excel 时发现目标系统要求税率字段不能为空",
      "change": {
        "section": "Troubleshooting",
        "action": "append",
        "target": "body",
        "content": "生成发票 Excel 前，必须确认税率字段；如果用户未提供税率，应先询问，不要默认填 0。"
      },
      "applied": false,
      "score": 0.6,
      "usage_stats": {
        "times_presented": 0,
        "times_used": 0,
        "times_positive": 0,
        "times_negative": 0
      }
    }
  ]
}
```

你不需要手写这些字段。

日常只建议编辑经验内容本身，也就是 `change.content`。像 `id`、`source`、`timestamp`、`score`、`usage_stats`、`applied` 这类字段由系统维护，手动改动容易破坏生命周期管理。

## 系统如何发现值得演进的信号？

在线自演进的输入来自真实使用，而不是离线拍脑袋。

典型信号包括四类。

### 1. 工具或脚本执行失败

例如：

- API 超时。
- 命令不存在。
- 权限不足。
- 文件格式不符合预期。
- 某个脚本在特定输入下报错。

这类信号适合沉淀为“排障经验”或“前置检查”。

例如：

```text
经验：调用导出脚本前，先检查输出目录是否存在；不存在时先创建目录。
```

### 2. 用户纠错和偏好反馈

用户说“不对”“应该这样”“你理解错了”“以后先问我”，往往比错误日志更有价值。

例如：

```text
用户：不对，我说的是技术调研邮件，不是论文摘要。
```

这可以演进为：

```text
当用户要求“技术调研邮件”时，输出对象是组内研发同学，应包含背景、趋势、讨论问题和行动建议，而不是只做论文摘要。
```

### 3. 任务执行中的有效经验

有时没有失败，但 Agent 在解决问题时探索出更好的路径。

例如：

- 某个接口不稳定，先做轻量探测再批量请求更稳。
- 大文件处理时先抽样，再决定是否分块。
- 多论文调研时先做粗筛，再对候选论文精读。

这些经验可以提高后续任务效率。

### 4. 团队协作中的新约束

对 Swarm Skill 来说，信号还可能来自团队层：

- 需要新增一个角色。
- 某个角色职责太宽，需要拆分。
- Leader 汇总时经常遗漏反方观点。
- SwarmFlow 某个阶段需要重试、预算限制或人工确认。

这类经验用于改进已有 Swarm Skill 的协作规则和流程。

## 自动演进：系统先观察，再生成候选经验

理解了演进对象和信号来源之后，再来看自动演进流程就很简单了。

在 JiuwenSwarm 配置页中，可以开启“自动检测可演进信号”。开启后，系统会在对话和工具执行后观察是否出现值得沉淀的经验。

自动演进大致分为五步：

```text
执行已有 Skill / Swarm Skill
        ↓
观察工具结果、用户反馈和团队轨迹
        ↓
识别可演进信号
        ↓
生成候选经验补丁
        ↓
等待用户确认后写入经验库
```

自动演进不是创建新 Skill。它要求当前任务已经关联到某个已有 Skill 或 Swarm Skill，并围绕这个已有能力补充经验。

例如：

- `invoice-xlsx` 在生成发票表时漏问税率，系统可以建议补充“生成前必须确认税率”。
- `paper-reading` 在用户纠正后发现“技术调研邮件”不是“论文摘要”，系统可以建议补充输出对象和结构偏好。
- `research-swarm` 在执行中发现缺少实验审计视角，系统可以建议补充团队层经验。

如果系统发现的是“当前还没有合适 Skill，需要创建一个新的 Skill 或 Swarm Skill”，那属于自动建议创建流程，不属于这里讨论的在线演进已有能力。

## 用户确认：自动生成经验不会静默落盘

在线自演进最重要的安全边界是：**系统自动生成经验和真正写入之间，通常会经过用户确认。**

系统检测到信号后，会先生成候选经验，然后弹出类似这样的审批：

```text
Skill 'xlsx' 演进生成了新经验：

- 目标：body
- 章节：Troubleshooting

生成发票 Excel 前，必须确认税率字段；如果用户未提供税率，应先询问，不要默认填 0。

选项：
- 接收
- 拒绝
```

如果你接收，经验才会写入经验库。

如果你拒绝，本次经验会被丢弃。

需要注意，不同治理命令的确认边界不完全相同：

- 自动检测信号生成的新经验，通常会先让用户接收或拒绝。
- 普通 Skill 的 `/evolve_simplify` 通常会先列出整理动作，再让用户确认是否执行。
- Team / Swarm Skill 的整理路径在当前版本中可能会直接执行并返回结果。
- `/evolve_rebuild` 是用户主动发起的重建任务，会先归档旧版本并生成后续任务提示。
- `/evolve_rollback` 是用户主动执行的回滚命令，不再额外弹出审批。

因此，更准确地说，openJiuwen 的自演进不是“Agent 悄悄改自己”，而是把不同阶段显式暴露给用户：

```text
真实使用产生信号
        ↓
系统生成候选经验
        ↓
用户确认（自动经验路径）
        ↓
写入经验库
        ↓
下次按需加载
```

## 手动触发：当你明确知道要改什么

自动演进适合让系统在后台观察信号。手动触发适合另一种情况：你已经明确知道某个已有 Skill 或 Swarm Skill 应该怎么改进。

### 立即触发已有 Skill 演进

使用：

```bash
/evolve <skill_name> [user_query]
```

例如：

```bash
/evolve xlsx 创建发票文件前需要向我确认字段和税率，不要自动猜测
```

在 Team 模式下，也可以对团队技能提出演进意图，但通常需要明确写出希望改进什么：

```bash
/evolve research-swarm 增加实验审计角色，专门检查论文实验软肋
```

### 查看和整理经验

查看某个 Skill 已积累的经验：

```bash
/evolve_list <skill_name> [--sort score]
```

当经验变多、重复、过时或质量参差不齐时，可以让系统生成整理方案：

```bash
/evolve_simplify <skill_name> [user_intent]
```

例如：

```bash
/evolve_simplify xlsx 合并重复的导出失败经验，删除不再适用的旧规则
```

普通 Skill 的整理方案通常会先弹出确认；Team / Swarm Skill 的整理路径在当前版本中可能会直接执行并返回结果。因此，整理前建议先用 `/evolve_list` 看清经验库。

### 重建与回滚

当一个 Skill 积累了大量高质量经验，希望把这些经验重新组织进 `SKILL.md`，可以使用：

```bash
/evolve_rebuild <skill_name> [user_intent]
```

例如：

```bash
/evolve_rebuild paper-reading 把高分经验整理进阅读流程，弱化低分经验
```

重建会发起一次后续重建任务。系统会先归档旧版本并准备重建上下文，真正的新版本生成依赖后续 Agent 继续执行。

需要回退时，可以使用：

```bash
/evolve_rollback <skill_name> [version]
```

如果不指定版本，系统会列出可回滚的归档版本。

## 运行时如何使用这些经验？

当某个 Skill 被调用时，系统不仅会读取原始 `SKILL.md`，也会读取它积累的演进经验。

但不是把所有历史经验一股脑塞进去。

系统会根据经验的目标位置和评分等信息，对经验进行加载、排序和展示。

高分、近期、经常被采纳的经验更容易出现在前面；低分、过时、重复或从未被采纳的经验会被降权，后续还可以被整理或删除。

用户体感上，它表现为：

- Agent 遇到同类错误时，更快想到已知处理方式。
- Agent 更能记住用户之前纠正过的偏好。
- Team 里的角色和协作约束会随着真实任务逐步完善。
- 不需要每次重新解释同一个坑。

这就是“越用越强”的实际含义。

不是模型参数变了，而是 Skill 的经验层变厚了、变准了。

---

## 经验如何评分？

经验不能无限增长。

如果所有记录都永久保留，经验库迟早会变成另一个垃圾堆：重复、过时、互相冲突，最终拖慢 Agent，甚至误导 Agent。

所以 openJiuwen 为每条经验维护质量分数。分数主要来自三个维度。

### 1. 有效性（Effectiveness）

这条经验被使用后，任务表现是否变好？

如果一条经验被采纳后解决了问题，它的有效性会提高。如果它导致错误或误导，分数会下降。

### 2. 使用率（Utilization）

这条经验展示给 Agent 后，Agent 是否真的用了？

如果一条经验多次被展示，但从未被采用，说明它可能不相关、太抽象或触发范围过宽。

### 3. 新鲜度（Freshness）

这条经验是否仍然适用于当前 Skill？

经验会随时间衰减。如果 Skill 主体已经重建或升级，旧经验也可能被降权，避免过时规则继续影响新版本。

系统会综合这些维度计算总分。默认理解即可：

> 高分经验优先注入，低分经验逐步降权，并进入整理候选。

## 生命周期管理：经验要有进有出

在线自演进不是“只进不出”。

一个健康的经验库应该像团队知识库一样：新问题不断沉淀，旧经验定期整理，过时内容可以删除，高价值内容可以固化。

### 查看：知道 Skill 学到了什么

使用：

```bash
/evolve_list <skill_name> [--sort score]
```

你可以查看某个 Skill 当前积累了哪些经验、分数如何、哪些经验更常被使用。

### 整理：合并重复、删除低质、精炼冗长

使用：

```bash
/evolve_simplify <skill_name> [user_intent]
```

系统会生成整理动作或直接执行整理，常见动作包括：

- `DELETE`：删除低质量或过时经验。
- `MERGE`：合并相似经验。
- `REFINE`：精炼单条经验。
- `KEEP`：保留高价值经验。

普通 Skill 的整理动作通常需要确认后才会执行；Team / Swarm Skill 的整理路径在当前版本中可能会直接执行。因此，执行整理前建议先用 `/evolve_list` 查看经验内容。

### 重建：把高价值经验重新组织进 Skill

当一个 Skill 经过长期使用，积累了很多高质量经验，就可以重建。

使用：

```bash
/evolve_rebuild <skill_name> [user_intent]
```

重建会发起一条后续重建任务，核心过程包括：

1. 归档当前 `SKILL.md`。
2. 归档当前 `evolutions.json`，筛选高分经验作为重建上下文。
3. 生成后续任务提示，引导 Agent 使用 `skill-creator` 或 `swarmskill-creator` 重新组织 Skill 主体。
4. 在重建上下文准备完成后，当前活跃经验会被清空或重置，避免新旧经验重复注入。

重建不是一个“直接覆盖按钮”，也不是简单把所有经验拼到文档后面。它更像是一次有归档保护的重写任务：先保留旧版本，再把高价值经验重新结构化到合适的位置。

例如：

- 多条排障经验可以合并成一个清晰的 `Troubleshooting` 部分。
- 多条用户偏好可以整理成执行前检查清单。
- Swarm Skill 中多条团队协作经验可以沉淀为新角色、质量门或失败处理策略。

### 回滚：重建不满意可以退回

重建前系统会归档旧版本。

如果重建结果不满意，可以使用：

```bash
/evolve_rollback <skill_name> latest
```

或者先查看可回滚版本：

```bash
/evolve_rollback <skill_name>
```

再选择指定版本回滚。

这让“把经验固化进主文档”不再是一次不可逆操作。

## Skill 和 Swarm Skill 都能演进吗？

可以，但演进对象不同。

### Agent Skill 的演进

普通 Skill 的演进主要关注：

- 执行步骤是否缺少前置条件。
- 工具调用是否需要重试或降级。
- 用户偏好是否需要补充。
- 输出格式是否要更严格。
- 脚本或模板是否需要补充说明。

例如：

```text
以后生成发票 Excel 前，必须先确认税率字段。
```

### Swarm Skill 的演进

Swarm Skill 的演进除了成员 Skill 的经验，还会关注团队层：

- 是否需要新增、删除或拆分角色。
- Leader 是否需要更清晰的分工规则。
- 某个阶段是否需要质量门。
- 并行任务是否需要预算和超时约束。
- SwarmFlow 是否需要补充失败处理或人工确认节点。

例如：

```text
短视频团队在多平台分发时，应新增“平台运营”角色，负责标题和文案适配。
```

可以把它理解为双层进化：

- 成员层：每个 Teammate 的执行经验变多。
- 团队层：Leader 的分工、协作和管控能力变强。

## 自动创建和在线自演进的边界

在 openJiuwen 的 Skill 体系里，有两个容易混淆的概念：

- 自动建议创建 Skill / Swarm Skill。
- 在线自演进已有 Skill / Swarm Skill。

它们的边界很清楚。

### 自动建议创建：从无到有

当系统发现某次任务或团队协作形成了可复用模式，但当前还没有合适 Skill，它会建议创建一个新 Skill 或 Swarm Skill。

它回答的是：

> 这次跑通的新模式，要不要沉淀成一个新能力？

### 在线自演进：让已有能力变好

当一个已有 Skill 或 Swarm Skill 在真实使用中出现失败、偏差、重复经验或改进机会，系统会生成经验补丁。自动生成的新经验通常经用户确认后写入经验库；整理、重建和回滚则按对应命令的行为执行。

它回答的是：

> 这个已经存在的能力，要不要补充、修正、整理或重建？

这里讲的是后者。

## 一个完整例子：发票 Excel Skill 如何越用越稳

假设你有一个 `invoice-xlsx` Skill，用于生成系统可导入的发票 Excel。

### 第一次：发现问题

用户说：

```text
帮我生成一份发票导入表。
```

Agent 生成了文件，但导入系统时报错：税率字段不能为空。

用户纠正：

```text
不对，税率必须确认后再写，不能默认空着。
```

### 第二步：生成经验

系统识别到这是一个可演进信号，生成候选经验：

```text
生成发票 Excel 前，必须确认税率字段；如果用户未提供税率，应先询问，不要默认留空。
```

### 第三步：用户确认

系统弹出确认问题。用户选择“接收”。

经验写入 `invoice-xlsx` 的经验库。

### 第四步：下次按需加载

下次用户再说：

```text
帮我生成发票导入表。
```

Agent 调用 `invoice-xlsx` 时会加载这条经验，于是先问：

```text
请确认本次发票税率是多少？
```

### 第五步：经验继续被评分

如果这条经验多次帮助任务成功，它的分数会上升。以后会更优先被注入。

如果它不再适用，分数会下降，后续可以被整理或删除。

### 第六步：长期重建

当 `invoice-xlsx` 积累了很多类似经验，例如税率、日期、抬头、导入字段、输出目录等，用户可以执行：

```bash
/evolve_rebuild invoice-xlsx 把高分经验整理成发票生成前检查清单
```

系统会归档旧版 `SKILL.md`，把高价值经验重新组织进主流程。

## 常见问题

### Q1：自演进会不会偷偷改我的 Skill？

不会。

自动检测或 `/evolve` 生成的新经验通常需要用户确认，接收后才会写入经验库。`/evolve_simplify`、`/evolve_rebuild`、`/evolve_rollback` 是用户主动发起的治理命令，行为边界各不相同：整理前建议先查看经验库，重建前会归档旧版本，回滚会按你指定的归档版本执行。

### Q2：演进经验和 `SKILL.md` 是什么关系？

`SKILL.md` 是基础能力定义，`evolutions.json` 是真实使用中积累的经验层。

运行时会把相关经验和原始 Skill 一起提供给 Agent。只有当你执行重建时，高价值经验才会被重新组织进 `SKILL.md`。

### Q3：经验越多越好吗？

不是。

经验过多会增加上下文负担，也可能产生冲突。所以系统会维护评分，并提供 `/evolve_simplify` 来合并、删除和精炼经验。

### Q4：什么时候该用 `/evolve`？

当你明确知道某个已有 Skill 或 Swarm Skill 需要改进时。

例如：

- “以后先问我，不要自己猜。”
- “遇到这个错误时应该先检查环境变量。”
- “这个团队需要一个专门审实验软肋的角色。”

### Q5：什么时候该用 `/evolve_rebuild`？

当某个 Skill 已经积累了较多高质量经验，而且这些经验已经稳定到可以进入主流程时。

如果只是补充一两条经验，不需要重建。

### Q6：Skill 创建、Swarm Skill 创建和在线自演进是什么关系？

Agent Skills 解决“如何创建单 Agent 能力”。

Swarm Skills 解决“如何创建多 Agent 团队能力”。

在线自演进解决“这些能力创建之后，如何在真实使用中持续变好”。

## 总结

openJiuwen 的 Skill 在线自演进不是一句抽象口号，而是一套完整闭环：

```text
真实任务执行
  → 发现失败、纠错、偏好和协作信号
  → 生成经验补丁
  → 用户确认（自动经验路径）
  → 写入经验库
  → 运行时按需加载
  → 使用效果评分
  → 整理、重建、回滚
```

这套机制让 Agent 不再只是调用一份静态说明书，而是在每一次真实使用中积累经验。

它的价值不是“让系统自己乱改自己”，而是让能力演进变得可见、可控、可评估、可回退。

这也是 openJiuwen 自演进体系的核心：让 Skills 和 Swarm Skills 从一次性配置，变成长期成长的能力资产。
