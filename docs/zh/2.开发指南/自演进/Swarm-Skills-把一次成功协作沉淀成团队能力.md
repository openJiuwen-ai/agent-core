# Swarm Skills：把一次成功协作沉淀成团队能力

Agent Skills 可以把单个 Agent 的专业知识、操作流程、脚本和模板沉淀成可复用能力。

但真实任务经常不是“一个 Agent 学会一套流程”就能解决。

例如：

- 代码审查时，需要可读性、性能、安全三个视角同时看，同一个 Agent 轮流扮演多个角色，观点很容易同源。
- 金融分析时，需要宏观、行业、财务、风险、投资建议多个专家并行研判，再统一汇总。
- 论文分享时，需要先读论文、再做方法拆解、再生成材料、最后写邮件，阶段之间有明确交接。
- 大型 PPT 生成时，需要规划章节、并行生成、统一审校，如果全靠一个 Leader 临场调度，很容易乱。

这些任务的难点不是“Agent 没有某个知识点”，而是“协作方式本身需要被沉淀”。

Swarm Skills 解决的就是这个问题。

一句话理解：

> Agent Skill 沉淀的是单个 Agent 的专业能力；Swarm Skill 沉淀的是一支多角色 Agent 团队的协作能力。

---

## 最快上手：一句话生成一支团队

使用 Swarm Skills 的最短路径不是手写角色文件，也不是自己编排每个 Agent，而是直接使用 `swarmskill-creator`。

你只需要描述“我想要一支什么团队，帮我完成什么任务”。

例如：

```text
帮我创建一个 Swarm Skill，用于代码安全审计。
我希望团队里至少有可读性 reviewer、性能 reviewer 和安全 reviewer。
他们先独立审查，再由一个总结角色整合问题，按严重程度输出报告。
```

或者：

```text
帮我创建一个 Swarm Skill，用于组内论文分享。
团队需要先阅读论文，再分析方法和实验，最后生成一封适合发给研发团队的分享邮件。
如果流程能提前确定，请生成 SwarmFlow 工作流脚本。
```

通常你会经历这样一个过程：

1. 你用自然语言描述目标任务。
2. `swarmskill-creator` 判断这个任务是否真的需要多 Agent 团队。
3. 如果适合，它会选择合适的协作方式和输出形态。
4. 它设计角色、分工、协作流程、约束规则和依赖资源。
5. 它生成标准 Swarm Skill 文件夹，并进行结构校验。
6. 下次遇到类似任务时，Team 模式可以直接调用这支团队。

因此，用户真正需要掌握的不是“怎么手写团队目录”，而是“什么时候该把一次协作沉淀成团队技能”。

---

## 什么是 Swarm Skill？

Swarm Skill 是 Agent Skills 标准的多角色扩展。

普通 Agent Skill 描述的是“一个 Agent 应该如何完成一个任务”。Swarm Skill 描述的是“一支 Agent 团队应该如何协作完成一个任务”。

它仍然是一个文件夹，入口仍然是 `SKILL.md`，但会进一步描述：

- 这支团队有哪些角色。
- 每个角色负责什么、不负责什么。
- 角色之间如何协作、交接和汇总。
- 遇到失败、超时、冲突或资源超限时如何处理。
- 每个角色可以使用哪些 Skills、工具或外部资源。

一个典型 Swarm Skill 结构如下：

```text
my-swarm/
├── SKILL.md              # 团队入口：名称、描述、角色列表、触发条件
├── roles/                # 每个角色的职责、边界、输出格式
│   ├── researcher.md
│   ├── reviewer.md
│   └── editor.md
├── workflow.md           # 团队如何协作：并行、串行、汇总、质量门
├── bind.md               # 团队约束：预算、超时、失败处理、降级策略
├── dependencies.yaml     # 角色依赖的 Skills、工具和外部能力
└── scripts/
    └── workflow.py       # 可选：SwarmFlow 可执行编排脚本
```

这个结构你了解即可。实际创建时，推荐交给 `swarmskill-creator` 自动生成。

在 openJiuwen 的代码和部分历史文档里，你也可能看到 `Team Skill` 这个名字。面向用户理解时，可以把它们看成同一类能力：都是把多 Agent 协作模式沉淀为可复用的团队技能。本文统一使用产品口径 `Swarm Skill`。

---

## 为什么需要 Swarm Skill？

不是所有复杂任务都需要多 Agent。Swarm Skill 只有在单 Agent 结构性吃力时才有价值。

最典型的是三类场景。

### 1. 视角同源：一个 Agent 很难真正“左右互搏”

让同一个 Agent 同时当“方案设计者”和“挑刺审稿人”，看起来有两个角色，实际上常常共享同一套分析先验。

它能表演出不同语气，却不一定能形成真正独立的判断。

适合 Swarm Skill 的例子：

- 代码审查：可读性、性能、安全三个角色独立审查。
- 产品方案评审：用户视角、商业视角、工程视角分别判断。
- 论文审稿模拟：支持者、反对者、实验审计者分别给意见。

这类任务的关键不是多写几段 prompt，而是让不同角色保持边界和独立立场。

### 2. 并行受限：多个子任务本来就可以同时做

有些任务天然可以拆成多个独立视角。

如果只让一个 Agent 串行处理，它不仅慢，还容易让前一个子任务的上下文干扰后一个子任务。

适合 Swarm Skill 的例子：

- 多平台竞品分析。
- 多论文并行阅读。
- 金融尽调中的宏观、行业、财务、风险并行分析。
- 大型文档的多章节并行生成。

这类任务的关键是并行分解和最终整合。

### 3. 阶段耦合：流程有明确交接和质量门

有些任务不是并行，而是有清楚阶段。

例如“构思 → 撰稿 → 编辑 → 审核”，每一步都需要承接前一步产出，但又不能混在一起。单 Agent 很容易一边构思一边编辑，一边编辑一边重写目标，导致阶段边界模糊。

适合 Swarm Skill 的例子：

- 论文分享：阅读 → 方法拆解 → 材料整理 → 邮件生成。
- 短视频生产：选题 → 脚本 → 分镜 → 标题文案 → 平台适配。
- 事故复盘：信息收集 → 时间线还原 → 根因分析 → 改进项制定。

这类任务的关键是阶段交接、质量门和失败处理。

## Swarm Skill 的两种协作方式

Swarm Skill 不是“越自动越好”，也不是“所有协作都要写成代码”。

openJiuwen 的判断很清楚：

> 编排能提前确定，就交给系统稳定执行；编排需要临场讨论，就保留开放协作。

因此，同一个 Swarm Skill 可以采用两种主要协作方式。

### 协作方式一：开放协作型 Swarm Skill

这种方式不强制生成可执行脚本，重点沉淀角色、规则、边界和协作方式。

它适合编排本身会随上下文变化的任务。

例如：

- 多专家圆桌研讨。
- 方案评审和交叉质询。
- 战略讨论。
- 创意策划。
- 狼人杀这类多角色互动游戏。

这类任务里，角色和大致阶段是确定的，但“谁回应谁、谁质疑谁、观点如何流动”需要临场发生。

强行写死流程，反而会削弱协作质量。

开放协作型 Swarm Skill 更像一份“团队协作说明书”：告诉 Leader 应该组织哪些角色、如何保持角色边界、什么时候隔离思考、什么时候互相讨论、最后如何汇总。

### 协作方式二：SwarmFlow 可执行编排

这种方式会生成 `scripts/workflow.py`，用 SwarmFlow 把确定的协作流程写成可执行工作流。

它适合编排可以提前确定的任务。

例如：

- 多篇论文逐篇分析，再汇总成报告。
- 金融分析中多个维度并行研判，再统一给出投资建议。
- 批量资料收集、整理、生成邮件。
- 大型 PPT 按章节并行生成，再统一合并。

SwarmFlow 的核心思路是：

> 编排归系统，智能归 Agent。

谁先做、谁并行、谁把结果交给谁、什么时候汇总、失败如何处理，这些由系统稳定执行。每个节点内部怎么理解、怎么推理、怎么产出，则交给对应 Agent。

这让 Swarm Skill 从一份“协作说明书”，进一步长出一条“可执行工作流”。

## swarmskill-creator 的三种输出形态

开放协作和 SwarmFlow 是用户理解层的两种协作方式。落到 `swarmskill-creator` 的产物上，会对应三种输出形态。

### 1. Markdown spec：默认团队规范

这是默认形态，用于创建一个完整的、可阅读的团队协作能力包。

它通常包含：

- `SKILL.md`：团队入口、触发条件、角色列表和文件索引。
- `roles/`：每个角色的职责、边界和输出格式。
- `workflow.md`：团队协作流程、并行或串行关系、质量门。
- `bind.md`：资源约束、失败处理、降级策略。
- `dependencies.yaml`：角色依赖的 Skills 和工具。

这种形态适合开放协作型任务，例如方案评审、多专家研讨、创意策划和交叉审查。

### 2. Script-only SwarmFlow：最小可执行工作流

当用户明确要的是“工作流”“SwarmFlow”“可执行编排”或 `workflow.py`，但不需要完整团队说明书时，`swarmskill-creator` 可以生成最小形态：

```text
my-workflow-swarm/
├── SKILL.md
└── scripts/
    └── workflow.py
```

这种形态只保留一个最小 `SKILL.md` 入口和一个 `scripts/workflow.py`。角色提示词、结构化输出 schema、阶段定义和编排逻辑都写在脚本中。

它适合流程非常确定、目标是“先跑起来并可复用”的场景，例如批量处理论文、批量生成报告、固定步骤的办公自动化。

### 3. Markdown spec + SwarmFlow：完整团队规范加可执行脚本

有些复杂场景既需要完整团队说明，也需要可执行编排。

例如，一个企业级投研 Swarm Skill 可能既要说明宏观分析师、行业分析师、风控专家的角色边界，也要提供一条固定执行的 SwarmFlow 工作流。

这时 `swarmskill-creator` 可以同时生成：

- 标准团队规范文件：`SKILL.md`、`roles/`、`workflow.md`、`bind.md`、`dependencies.yaml`。
- 可执行编排脚本：`scripts/workflow.py`。

用户不需要自己判断文件结构，只要把目标说清楚。

## 如何用 swarmskill-creator 创建 Swarm Skill？

`swarmskill-creator` 是 Swarm Skill 的配套创作工具。

它本身也是一个标准 Agent Skill，专门用于创建、转换或修改 Swarm Skills。你可以把它理解为“团队技能自动生成专家”。

它支持三类操作。

### 创建：从一句需求生成一支团队

适合你还没有现成团队技能，只知道自己想完成一个复杂任务。

示例：

```text
帮我创建一个 Swarm Skill，用于代码审查。
我希望它能同时关注可读性、性能和安全问题。
不同 reviewer 先独立输出，再由 lead reviewer 汇总成最终报告。
```

`swarmskill-creator` 会判断这个场景是否真的需要团队。如果单 Agent Skill 已经足够，它应该建议你不要过度编排。

如果适合团队，它会继续设计：

- 应该有哪些角色。
- 每个角色的立场、成功标准和边界。
- 应该并行还是串行。
- 哪些阶段需要质量门。
- 哪些本地 Skills 或工具可以分配给对应角色。
- 是否需要 SwarmFlow 可执行脚本。

### 转换：把单 Agent Skill 升级成团队技能

有些单 Agent Skill 已经包含多个隐含角色。

例如一个代码审查 Skill 里同时写着：

- 检查可读性。
- 检查性能。
- 检查安全。
- 最后输出综合报告。

这时可以让 `swarmskill-creator` 将它转换成 Swarm Skill：

```text
把 @code-review 这个单 Agent Skill 转换成 Swarm Skill。
保留原有审查标准，但拆成可读性、性能、安全三个 reviewer。
```

转换的意义不是把文件变多，而是把原来混在一个 Agent 里的多个职责拆开，让角色边界更清晰。

### 修改：持续调整已有团队

Swarm Skill 创建后也可以继续调整。

例如：

```text
给 short-video-swarm 增加一个平台运营角色，
专门负责把标题和文案适配到抖音、小红书和 B 站。
```

或者：

```text
把这个投研 Swarm Skill 改成 SwarmFlow 版本，
让宏观、行业、财务三个分析并行执行，最后统一汇总。
```

`swarmskill-creator` 会尽量只修改受影响的文件，并重新校验结构。

## 怎样描述需求，才能生成更好的 Swarm Skill？

你不需要懂目录结构，但需要说清楚“为什么需要一支团队”。

可以使用下面这个模板：

```text
帮我创建一个 Swarm Skill，用于【任务场景】。
单 Agent 当前的问题是【视角同源 / 不能并行 / 阶段容易混乱】。
我希望团队包含【关键角色或专业视角】。
协作方式希望是【独立并行 / 顺序流水线 / 先独立后讨论 / 固定工作流】。
最终输出需要【报告 / 文件 / 邮件 / 决策建议】。
如果流程能提前确定，请生成 SwarmFlow 工作流脚本。
```

示例：

```text
帮我创建一个 Swarm Skill，用于技术方案评审。
单 Agent 的问题是经常只从实现者角度看方案，缺少攻击性审查。
我希望团队包含架构师、SRE、安全审查员和产品代表。
他们先独立评审，再进行交叉质询，最后由评审主席汇总风险和改进建议。
最终输出按“阻塞问题、重要建议、可选优化、开放问题”组织。
```

如果你想要可执行工作流，可以说得更明确：

```text
帮我创建一个 SwarmFlow，用于批量阅读论文并生成组内分享邮件。
流程固定为：逐篇论文并行解析 → 汇总关键观点 → 生成分享提纲 → 起草邮件。
请生成可执行的 workflow.py。
```

这里的关键是：不要只说“帮我建一个强大的团队”。要说明单 Agent 哪里不够、团队应该如何补足。

## Team 模式下也可以基于轨迹建议沉淀

除了主动使用 `swarmskill-creator`，系统也可以基于真实 Team 执行轨迹提出 Swarm Skill 沉淀建议。

这对应的是“用过的成功协作，长成可复用团队”。

例如你第一次执行短视频创作任务时，并没有现成 Swarm Skill。Leader 临时组建了一支团队：

- 选题策划角色。
- 脚本创作角色。
- 分镜设计角色。
- 视觉风格角色。
- 标题文案角色。

任务完成后，系统发现这次协作不是一次性闲聊，而是一套可复用的团队模式。于是它会询问：

```text
我检测到多 Agent 协作模式可能值得创建为团队技能。是否创建？

选项：
- 创建
- 跳过
- 自定义指令：请描述你希望如何沉淀这个团队技能
```

只有当你选择“创建”或补充自定义指令后，系统才会调用 `swarmskill-creator`，结合当前团队轨迹生成新的 Swarm Skill。

这点和 Agent Skill 自动建议创建类似，但对象不同：

- Agent Skill 自动建议创建，沉淀的是单 Agent 的可复用工作模式。
- Swarm Skill 自动建议创建，沉淀的是多 Agent 的可复用协作模式。

当前 Team 模式下的自动建议沉淀主要关注是否出现了实际多成员协作，以及这次协作是否已经使用了已有团队技能。如果已经是基于已有 Swarm Skill 执行，系统会避免重复建议创建。

如果你希望启用这类能力，可以在 JiuwenSwarm 的配置页打开“自动建议创建新技能”。具体配置入口可能随版本变化，推荐优先以产品配置页为准。

这里讨论的是“从一次成功团队协作中，经用户确认创建新的 Swarm Skill”。它不是修改已有技能；已有 Skill 或 Swarm Skill 如何持续改进，属于在线自演进机制的范围。

## 什么时候不该使用 Swarm Skill？

Swarm Skill 很强，但不是所有任务都该上团队。

以下情况不建议使用：

- 单 Agent Skill 已经能稳定完成任务。
- 任务没有明显角色分工，只是希望“多几个 Agent 更聪明”。
- 子任务之间强依赖、难以并行，也没有明确阶段交接。
- 任务很小，多角色带来的成本超过收益。
- 你还没有跑通过一次有效流程，过早沉淀会固化错误协作方式。

一个实用判断是：

> 如果你说不清楚“为什么一个 Agent 做不好”，就先不要创建 Swarm Skill。

可以先用普通 Agent 或 Agent Skill 跑通，再看是否需要团队化。

## Swarm Skill、Agent Skill、SwarmFlow 的关系

这三个概念可以这样理解：

### Agent Skill：一个人的专业手册

它回答的是：

> 一个 Agent 遇到某类任务时，应该如何做？

适合单角色、稳定流程、专业知识沉淀。

### Swarm Skill：一支团队的作战手册

它回答的是：

> 多个 Agent 遇到某类复杂任务时，应该如何分工、协作和汇总？

适合多视角、并行分解、阶段交接和团队协作。

### SwarmFlow：团队协作中的可执行编排

它回答的是：

> 如果协作流程可以提前确定，系统能不能稳定执行这条流程？

适合确定性强、步骤清晰、需要并行、流水线、阶段可观测的任务。

所以，SwarmFlow 不是另起炉灶的新概念。它是 Swarm Skill 中负责“可执行编排”的那部分能力。

## 一个完整例子：从临时团队到 Swarm Skill

假设你第一次让 JiuwenSwarm 做“多论文技术调研”：

```text
帮我调研最近 Agent 自演进方向的 5 篇论文，
总结技术路线、实验软肋和对 openJiuwen 的启发，
最后写一封适合发给组内研发同学的分享邮件。
```

第一次执行时，Leader 可能临时组织多个成员：

- 论文检索员：负责找到相关论文。
- 方法分析员：负责拆解技术路线。
- 实验审计员：负责检查实验软肋。
- 工程转化员：负责提炼对 openJiuwen 的启发。
- 邮件撰写员：负责生成分享邮件。

如果这次协作跑通了，后续就不应该每次都重新临场组队。

你可以主动说：

```text
基于刚才这次协作，帮我创建一个 Swarm Skill。
以后用于多论文技术调研和组内分享邮件生成。
保留论文检索、方法分析、实验审计、工程启发、邮件撰写这几个角色。
如果适合，请把固定流程生成 SwarmFlow。
```

如果开启了自动建议创建，系统也可能在 Team 任务完成后主动询问是否沉淀。

创建完成后，下次你只需要说：

```text
用多论文调研团队，帮我分析最近的 Agent Memory 论文。
```

Team 模式就可以直接加载这支团队，而不是重新发明一次协作流程。

这就是 Swarm Skills 的核心价值：

> 让一次成功的团队协作，变成下一次可直接调用的团队能力。

## 常见问题

### Q1：Swarm Skill 和普通 Agent Skill 有什么区别？

普通 Agent Skill 面向单个 Agent，沉淀的是专业知识和流程。

Swarm Skill 面向多 Agent 团队，沉淀的是角色分工、协作流程、约束规则和工具依赖。

### Q2：我需要手写 `roles/`、`workflow.md` 和 `bind.md` 吗？

不需要。

推荐优先使用 `swarmskill-creator`。你只需要描述任务目标、角色需求和协作方式。它会生成对应文件并进行校验。

### Q3：所有 Swarm Skill 都需要 `workflow.py` 吗？

不需要。

如果任务需要开放讨论、交叉质询或临场协商，保留 Markdown 形式的团队协作规范更合适。

只有当协作流程可以提前确定时，才建议生成 SwarmFlow 脚本。

### Q4：我应该直接创建 Swarm Skill，还是先跑一次 Team？

两种都可以。

如果你已经很清楚团队结构和流程，可以直接让 `swarmskill-creator` 创建。

如果你还不确定怎么分工，可以先让 Team 模式临时协作一次。跑通后，再基于轨迹沉淀成 Swarm Skill。

### Q5：为什么系统会跳过自动建议创建？

常见原因包括：

- 本次任务没有形成多成员协作。
- 已经使用了现有 Swarm Skill，不需要重复创建。
- 协作模式太临时，不具备复用价值。
- 用户没有确认创建。

自动建议创建的目标不是把所有任务都保存下来，而是识别真正值得复用的团队模式，并在用户确认后再沉淀。

## 相关主题

Skill 在线自演进机制关注的是：

> 当一个 Skill 或 Swarm Skill 已经存在，它如何在真实使用中持续变好？

它会涉及补丁机制、用户确认、经验记录、生命周期管理，以及为什么 openJiuwen 的自演进不是“偷偷改文件”，而是透明、可控、可回收的持续优化闭环。
