# Agent Skills：让 Agent 沉淀你的专业能力

很多用户第一次使用 openJiuwen 或 JiuwenSwarm 时，最常见的问题不是“模型够不够强”，而是：

- 我能给 Agent 增加哪些能力？
- 我怎样把自己的工作方法交给 Agent？
- 为什么同样的要求，每次都要重新解释一遍？
- Agent 跑通了一次复杂流程后，能不能下次直接复用？

Agent Skills 解决的就是这个问题。

它不是一个重型插件系统，也不是要求用户写代码的开发框架。更准确地说，Agent Skills 是一种轻量级、开放的能力封装格式：把专业知识、操作步骤、脚本、模板和参考资料放进一个标准文件夹，让 Agent 在需要时自动读取并执行。

一句话理解：

> Agent Skill 是给 Agent 使用的“专业操作手册 + 可复用资源包”。它让一次经验不再停留在一次对话里，而是沉淀成以后可以反复调用的能力。

---

## 最快上手：你不需要手写 Skill

使用 Agent Skills 的最短路径不是从零写 `SKILL.md`，而是直接让 `skill-creator` 帮你创建。

你只需要告诉它：你想沉淀什么能力、这个能力什么时候该触发、执行时有哪些偏好和边界。

例如：

```text
帮我创建一个 Skill，用于每周生成项目周报。
它需要先整理本周完成事项、风险和下周计划，
然后按照“进展、风险、依赖、下周重点”的结构输出。
语气要适合发给研发管理者。
```

或者：

```text
帮我创建一个 Skill，用于处理发票 Excel。
每次写入文件前必须先确认发票抬头、金额、税率、日期和导入系统字段。
如果字段缺失，要先问我，不要自己猜。
```

通常你会经历这样一个过程：

1. 你用自然语言描述想要的能力。
2. `skill-creator` 判断这个需求是否适合沉淀成 Skill。
3. 它生成标准 Skill 文件夹，包括入口说明和必要资源。
4. 你确认或补充要求。
5. 这个 Skill 被保存到技能目录。
6. 下次遇到相关任务时，Agent 会根据 Skill 描述自动判断是否调用。

因此，用户真正需要掌握的不是“怎么手写一个文件夹”，而是“怎么把自己的工作方法说清楚”。

## 什么是 Agent Skills？

Agent Skills 是一种轻量级、开放的格式，用于结合专业知识和工作程序性流程来扩展 AI Agent 能力。

Skills 的核心是一个包含 `SKILL.md` 文件的文件夹。该文件包含元数据（名称和描述必填）以及指示 Agent 如何执行特定任务的指令。Skills 还可以捆绑脚本（Scripts）、模板（Templates）和参考资料（References）。

一个典型 Skill 目录如下：

```text
my-skill/
├── SKILL.md          # Required: instructions + metadata
├── scripts/          # Optional: executable code
├── references/       # Optional: documentation
└── assets/           # Optional: templates, resources
```

这个结构你了解即可。实际创建时，推荐交给 `skill-creator` 自动生成。

其中：

- `SKILL.md` 是入口文件，必须存在。它告诉 Agent 这个 Skill 叫什么、什么时候该用、具体怎么做。
- `scripts/` 放可执行脚本，适合处理确定性强、步骤固定、容易出错的操作。
- `references/` 放较长的背景资料、接口文档、规范说明，只有需要时才读。
- `assets/` 放模板、示例文件、表格样式、提示词片段等可直接复用的资源。

一个 Skill 可以很轻，也可以很深。

轻量时，它可能只有一个 `SKILL.md`，用于告诉 Agent“处理发票文件时先确认哪些字段”。复杂时，它可以包含脚本、模板、校验规则和长文档，帮助 Agent 稳定完成一套完整流程。

## 为什么不是把所有说明都写进 Prompt？

因为 Prompt 会越来越长，而 Agent 真正需要的信息通常只占其中一小部分。

如果把所有领域知识、操作规范、错误处理和模板都塞进系统提示词，短期看似方便，长期会带来三个问题：

- 上下文被无关内容挤占，Agent 反而变慢、变钝。
- 不同任务之间互相干扰，本来只需要处理表格，却读了一堆写 PPT 的规则。
- 知识难以维护，修改一个流程说明可能影响所有任务。

Agent Skills 的核心设计是“按需加载”，也就是渐进式披露。

Skills 利用渐进式披露高效管理上下文。这种方式既能让 Agent 保持快速工作，又能按需获取极深度的上下文信息。

### 发现（Discovery）

启动时，Agent 只加载每个可用技能的名称和描述，仅仅足够让它知道何时可能相关。

例如：

```yaml
name: invoice-xlsx
description: 处理发票 Excel 文件，生成、校验或修复发票表格时使用；在写入前必须确认字段、金额和税率要求。
```

此时 Agent 并不会读取完整 `SKILL.md`，更不会把所有参考文档塞进上下文。

### 激活（Activation）

当任务与某技能描述匹配时，Agent 会自主决定将完整的 `SKILL.md` 指令解读到当前上下文中。

例如用户说：

```text
帮我整理这批发票，生成一个可以导入系统的 Excel。
```

Agent 会发现这与 `invoice-xlsx` 的描述匹配，于是加载完整 Skill，读取里面的字段确认规则、表格生成步骤、常见错误和输出格式要求。

### 执行（Execution）

Agent 按照指令工作，并可根据需要加载引用的长篇文档，或执行捆绑的代码脚本。

例如：

- 如果 `SKILL.md` 里写明“复杂字段映射见 `references/schema.md`”，Agent 只有在需要字段映射时才读取该文件。
- 如果 `scripts/validate_invoice.py` 可以校验发票格式，Agent 可以直接执行脚本，而不需要把脚本全文读入上下文。
- 如果 `assets/template.xlsx` 是标准模板，Agent 可以基于模板生成输出，而不是每次重新猜测格式。

这就是 Agent Skills 的核心优势：平时很轻，必要时很深。

## 什么场景值得创建 Agent Skill？

不是所有任务都需要 Skill。

如果一个任务只做一次，或者一句自然语言就能稳定完成，就没有必要沉淀。Skill 的价值在于“复用”和“稳定”。

当你遇到以下情况，就值得创建 Skill。

### 经常重复解释同一套要求

例如：

- 每次生成 PPT 前都要说明公司模板、章节顺序和图表风格。
- 每次处理 Excel 都要提醒字段校验、命名规则和输出目录。
- 每次写技术调研都要强调摘要、竞品、风险和推荐结论的结构。

这些不是一次性需求，而是你的工作习惯。把它们沉淀成 Skill，Agent 以后会自动遵守。

### 任务有稳定的操作流程

例如：

- 下载文件 → 解析字段 → 校验格式 → 生成报告。
- 收集资料 → 去重 → 分类 → 写邮件。
- 读取日志 → 定位错误 → 复现问题 → 给出修复建议。

这类流程很适合写成 Skill，因为它不是单纯知识，而是可执行的工作程序。

### 任务需要专业知识或团队规范

例如：

- 法务合同审查标准。
- 数据分析口径。
- 论文阅读报告格式。
- 代码评审关注点。
- 内部系统的接口调用约定。

这些内容通常不在模型的通用知识里，或者即使模型知道，也不一定符合你的组织偏好。Skill 可以把“通用模型能力”校准到“你的专业方法”。

### 任务依赖脚本、模板或参考资料

如果一个任务需要固定脚本、模板文件、长规范文档，Skill 比 Prompt 更合适。

Prompt 适合告诉 Agent“怎么想”；Skill 还能告诉 Agent“去哪找资料、跑哪个脚本、用哪个模板”。

## 如何用 skill-creator 创建一个好 Skill？

`skill-creator` 的价值不是“替你生成一个文件”，而是帮助你把模糊经验变成可复用能力。

你可以按下面这个模板描述需求：

```text
帮我创建一个 Skill，用于【任务场景】。
它应该在【触发条件】时使用。
执行时需要遵守【步骤 / 规范 / 偏好】。
如果遇到【边界情况】，应该【处理方式】。
输出格式是【期望结果】。
```

示例：

```text
帮我创建一个 Skill，用于论文阅读报告。
它应该在我让 Agent 阅读论文、总结论文或生成顶会洞察时使用。
执行时需要先判断论文贡献，再分析方法、实验、创新点和可借鉴之处。
如果实验薄弱，要明确指出可能被审稿人攻击的地方。
输出要用中文，结构包括：一句话结论、核心创新、方法拆解、实验软肋、可借鉴点。
```

一个高质量 Skill 最关键的是两件事：

### 1. 描述要能帮助 Agent 准确触发

不推荐：

```yaml
description: 处理论文。
```

推荐：

```yaml
description: 当用户要求阅读、总结、批判或生成中文论文阅读报告时使用；重点分析论文贡献、方法、实验软肋、审稿风险和可借鉴点。
```

前者只是标签，后者才是触发条件和行为边界。

### 2. 指令要像操作手册，而不是宣传文案

不推荐：

```text
你要认真处理用户需求，保证输出质量。
```

推荐：

```text
处理论文阅读任务时：
1. 先用一句话判断论文要解决的核心问题。
2. 再拆解方法，包括输入、关键模块、训练或推理流程。
3. 单独列出实验可能被质疑的地方。
4. 最后给出对 openJiuwen 或 Agent 自演进研究可借鉴的点。
```

Skill 是给 Agent 执行的，不是给人看的宣传稿。越可执行，越稳定。

## 系统也可以在使用过程中建议创建 Skill

有些 Skill 不是一开始设计出来的，而是在真实任务中“长出来”的。

例如你让 Agent 完成一次复杂调研，它搜索资料、筛选论文、整理观点、生成邮件，最后形成一套很顺的流程。这个执行过程本身就有复用价值。下次你做类似调研时，不应该从零再来一遍。

openJiuwen 可以在使用过程中观察任务轨迹，判断是否出现了值得沉淀的可复用模式。如果你开启了“自动建议创建新技能”，系统会在合适时机询问你：

```text
我检测到当前任务中存在可复用模式，可能值得创建一个新技能。是否创建？

选项：
- 创建
- 跳过
- 自定义指令：请描述你希望如何沉淀这个技能
```

只有当你选择“创建”或提供自定义指令后，系统才会调用 `skill-creator`，结合当前对话上下文和执行轨迹生成新 Skill。

这点很重要：

> 自动建议创建不是后台静默改你的工作区，而是“系统发现机会 → 用户确认 → skill-creator 创建”。

如果你希望启用这个能力，可以在 JiuwenSwarm 的配置页打开“自动建议创建新技能”。在当前默认配置文件中，对应能力位于 `react.evolution.skill_create`：

```yaml
react:
  evolution:
    skill_create: true
```

环境变量 `SKILL_CREATE=true` 会覆盖配置文件。具体配置入口可能随版本变化，推荐优先以产品配置页为准。

## 自动创建和在线自演进有什么区别？

这两个概念容易混淆，但边界很清楚。

### Skill 自动创建：从无到有

当系统发现某次任务跑出了一套可复用流程，但当前没有合适 Skill，它会建议创建一个新 Skill。

它回答的是：

> 这次新学到的模式，是否应该变成一个新能力？

### Skill 在线自演进：让已有 Skill 变好

当一个已有 Skill 在真实使用中出现失败、偏差、重复经验或需要整理时，系统会基于轨迹和用户反馈生成改进记录，经过审批后更新经验或重建 Skill。

它回答的是：

> 这个已有 Skill，是否应该被修复、补充或重组？

这里讨论的是 Agent Skill 的创建和使用。在线自演进的补丁机制、用户确认、经验生命周期和重建流程，属于“已有 Skill 如何持续改进”的主题。

## 一个完整例子：把技术调研沉淀成 Skill

假设你经常让 Agent 帮你做“技术调研邮件”。

第一次，你可以直接让 Agent 完成任务：

```text
帮我调研最近 Agent 自演进方向的新论文和开源项目，
整理成一封适合发给组内研发同学的技术分享邮件。
```

Agent 可能会搜索资料、筛选论文、整理观点、生成邮件。任务跑通后，如果你觉得这套流程以后还会反复使用，就可以对 `skill-creator` 说：

```text
基于刚才这次任务，帮我创建一个 Skill。
它以后用于技术分享前的调研邮件。
要求输出包括：背景、关键论文、趋势判断、风险、适合组内讨论的问题。
邮件语气要适合发给研发团队。
```

如果你开启了自动建议创建，系统也可能在识别到可复用流程后主动询问是否创建。

创建完成后，下次你再说：

```text
帮我准备一封关于多 Agent 协同的技术分享调研邮件。
```

Agent 就可以在发现阶段匹配到这个 Skill，并按里面沉淀的流程执行。

最终效果是：

> 你不需要每次重新教 Agent 怎么写。它会把跑通过的工作方式沉淀下来，下一次直接复用。

## 常见问题

### Q1：我需要先学会写 `SKILL.md` 吗？

不需要。

推荐先用 `skill-creator` 创建。你只需要描述目标、场景和偏好。生成后如果你想进一步优化，再编辑 `SKILL.md`。

### Q2：Skill 会不会让 Agent 变慢？

设计得好的 Skill 不会显著拖慢 Agent。

因为 Agent 启动时只加载名称和描述。完整 `SKILL.md`、参考资料和脚本都是按需加载。真正容易拖慢的是把大量无关说明塞进系统 Prompt，而不是使用 Skill。

### Q3：什么时候不该创建 Skill？

以下情况不建议创建：

- 任务只做一次。
- 一句话就能稳定完成。
- 只是临时信息，不具备复用价值。
- 已有 Skill 已经覆盖，不需要重复创建。
- 流程还没跑通过，沉淀后反而会固化错误经验。

### Q4：自动建议创建 Skill 会不会乱创建？

不会直接静默创建。

系统只是根据执行轨迹提出建议。真正创建前需要用户确认，确认后才会调用 `skill-creator` 生成 Skill。

### Q5：创建 Skill 时，我应该追求“大而全”吗？

不建议。

更好的方式是先创建一个小而准的 Skill，让它覆盖一个高频、明确、可复用的任务。后续在真实使用中再逐步补充经验。

Agent Skills 的价值不在于一次写出完美文档，而在于建立一个可持续沉淀的能力载体。

它让 Agent 从“每次重新理解任务”变成“逐渐掌握你的工作方法”。

## 相关主题

Swarm Skills 关注的是：

> 当一个 Skill 不再只是单个 Agent 的操作手册，而是一支多角色团队的协作能力包，应该如何创建、使用和沉淀？

相关内容包括 `swarmskill-creator`、Team 模式下的轨迹沉淀，以及开放协作与 SwarmFlow 两种形态。
