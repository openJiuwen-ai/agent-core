# Tool Optimizer（工具描述自演进）

`Tool Optimizer` 是 openJiuwen agent-core 的工具自演进模块。

它的核心目标是：
1. 给定工具描述和可调用函数；
2. 自动生成“可执行调用样本 + 指令/答案样本”；
3. 基于评估结果迭代改写描述；
4. 产出更清晰、更可执行、边界更明确的工具描述 JSON。


## 概述

该模块围绕“描述是否让模型更正确地调用工具”做优化，采用两阶段循环：
1. **Example 阶段**：生成并筛选高质量调用样本；
2. **Description 阶段**：结合正/负样本改写描述。

每轮结果会落盘，最终再经过 reviewer 做结构化、去冗余和一致性检查，输出符合原 schema 的最终描述。

## 快速开始

```python
import os
from dotenv import load_dotenv

from openjiuwen.agent_evolving.optimizer.tool.base import ToolOptimizerBase
from openjiuwen.agent_evolving.optimizer.tool.utils.default_configs import (
    default_config_desc,
    default_config_eg,
)
from openjiuwen.agent_evolving.optimizer.tool.utils.callable_fortest import (
    tool,
    gaode_map_mcp_generic as callable_func,
)

load_dotenv()

optimizer = ToolOptimizerBase(
    max_turns=2,
    config_eg=default_config_eg,
    config_desc=default_config_desc,
    llm_api_key=os.getenv("OPENAI_API_KEY", ""),
    path_save_dir="./tool_optimizer_results",
    tool_name=tool["name"],
)

result = optimizer.optimize_tool(tool, callable_func)
print(result)
```

## 输入输出协议

### 输入一：`tool`

最小必需字段：
- `name`: 工具名（字符串）
- `description`: 工具描述（推荐为 OpenAI function schema 的 JSON 字符串）

推荐 `description` 内容形态（示例）：
- 包含 `type/function/name/description/parameters/required` 等结构；
- 参数说明越准确，前几轮搜索越稳定。

### 输入二：`tool_callable`

签名约定：
- `tool_callable(arguments: Dict[str, Any]) -> Any`

注意：
- wrapper 会把生成的函数调用对象整体传入 callable（不是 kwargs 展开）；
- callable 抛异常会被记录为调用失败样本。

### 输出：`final_desc`

`optimize_tool` 返回 reviewer 处理后的描述 JSON（字典），结构对齐原始 schema，但文本内容已经过优化。

## 运行流程（`optimize_tool`）

每轮（`max_turns`）执行以下步骤：

1. **Example 阶段**
- 调用 `customized_pipeline("example", ...)`；
- 使用 `APICallToExampleMethod + BeamSearch` 生成 API call、用户指令、答案及评分；
- 结果写入 `examples/<tool_name>.json`。

2. **Description 阶段**
- 调用 `customized_pipeline("description", ...)`；
- 使用 `ToolDescriptionMethod + BeamSearch` 基于正/负样本改写描述；
- 结果写入 `descriptions/<tool_name>.json`。

3. **后处理阶段（最终）**
- `ToolDescriptionReviewer.process(steps=["clean", "cross_check", "translate"])`；
- `extract_schema` 提取原结构模板；
- `reviewer.format` 把优化文本重新映射到 schema，得到最终输出。

## 配置

默认配置位于 `utils/default_configs.py`：
- `default_config_eg`: Example 阶段
- `default_config_desc`: Description 阶段

关键参数说明：
- `gen_model_id`: 生成模型（例：`gpt-5-mini`）
- `eval_model_id`: 评估模型
- `beam_width`: 每层保留候选数
- `expand_num`: 每个节点扩展分支数
- `max_depth`: 搜索深度
- `top_k`: 返回候选路径数
- `num_workers`: 并行 worker 数
- `num_refine_steps`: Example 阶段指令迭代次数
- `num_feedback_steps`: 用于反思/改写的历史窗口
- `num_examples_for_desc`: Description 阶段读取样本数
- `score_eval_weight`: 是否叠加额外评估项（默认 0）

`ToolOptimizerBase` 额外参数：
- `max_turns`: 总迭代轮数（默认 5）
- `llm_api_key`: LLM key
- `path_save_dir`: 中间结果目录（默认 `./tool_optimizer_results`）
- `tool_name`: 用于负样本文件命名（`<tool_name>.json`）

## 核心组件

### 1) `ToolOptimizerBase`（`base.py`）
- 对外统一接口：`optimize_tool`
- 编排 Example/Description 两阶段
- 最终 reviewer 清洗与格式对齐

### 2) `customized_pipeline`（`utils/customized_pipline.py`）
- 根据 `stage` 选择方法类：
  - `example` -> `APICallToExampleMethod`
  - `description` -> `ToolDescriptionMethod`
- 统一接入 `BeamSearch`
- 统一结果保存逻辑

### 3) `APICallToExampleMethod`（`utils/toolcall_example_method.py`）
- 从描述生成可执行函数调用
- 调用工具获取真实输出
- 自动生成用户指令、答案与批量反思
- 输出可用于描述优化的高价值样本

### 4) `ToolDescriptionMethod`（`utils/description_example_method.py`）
- 读取正样本（成功）与负样本（失败）
- 先批判，再改写工具描述
- 使用 `SimpleEval` 回评估描述质量

### 5) `SimpleEval`（`utils/customized_eval.py`）
- 评估两部分：
  - 函数调用准确率（函数名 + 参数比对）
  - 输出有效性（LLM 打分或回退规则）
- 返回 `score_avg/score_std/fn_call_accuracy/output_effectiveness`

### 6) `ToolDescriptionReviewer`（`utils/customized_revivwer.py`）
- 清理冗余、交叉校验原描述信息、按需翻译
- 强制回填到目标 schema
- 保证输出结构可直接用于工具定义

## 产物与中间文件

默认输出目录：`./tool_optimizer_results`

- `examples/<tool_name>.json`
  - Example 阶段每次搜索历史
- `descriptions/<tool_name>.json`
  - Description 阶段候选描述与评分
- `<tool_name>.json`（可选）
  - 外部提供的负样本文件（Description 阶段优先读取）

## 开发与扩展建议

1. **替换模型**
- 改 `default_config_eg/default_config_desc` 的 `gen_model_id/eval_model_id` 即可。

2. **调整搜索开销**
- 降低 `beam_width/expand_num/max_depth/num_workers`。

3. **增强评估标准**
- 在 `SimpleEval` 中调整 `fn_call_weight/output_effectiveness_weight`；
- 或扩展 `_evaluate_output_effectiveness` 的打分提示词。

4. **引入外部负样本**
- 将负样本放到 `path_save_dir/<tool_name>.json`；
- `ToolDescriptionMethod.get_negative_examples` 会优先读取。

## 运行前置条件

1. Python 环境可导入 `openjiuwen`；
2. 可用的 OpenAI 兼容模型配置；
3. 设置 `OPENAI_API_KEY`（或通过构造参数传入 `llm_api_key`）。

示例：

```bash
set OPENAI_API_KEY=your_key
python base.py
```

## 已知限制

1. `customized_pipeline` 中 `fn_call_path` 分支尚未实现（会抛 `NotImplementedError`）；
2. 当前目录未提供自动化测试脚本，建议在接入新工具时先做小规模 smoke test；
3. 该模块依赖 LLM 质量与稳定性，建议固定模型版本并控制随机性。
