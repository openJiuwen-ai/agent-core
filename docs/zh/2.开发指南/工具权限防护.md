# 工具权限防护（Permissions）

本文说明 **openjiuwen** 中与「工具调用是否允许 / 是否需要人工确认」相关的机制，以及如何通过宿主（`ToolPermissionHost`）接入磁盘配置与产品策略。

它与 [安全护栏 Guardrail](./高阶用法/安全护栏Guardrail.md) 不同：Guardrail 偏 **内容风险检测**（提示注入、敏感数据等）；工具权限偏 **执行控制**（allow / ask / deny、Confirm 中断、落盘规则）。

---

## 1. 能力概览

| 组件 | 作用 |
|------|------|
| **`SecurityRail`** | 在模型调用前后注入安全相关提示或检测（提示层）。 |
| **`PermissionInterruptRail`** | 在 **`before_tool_call`** 上对**任意工具**做策略判定；`ask` 时走 Confirm 中断（执行层）。 |
| **`PermissionEngine`** | 加载 `permissions` 配置，评估 `allow` / `ask` / `deny`，供 Rail 与批量守卫使用。 |
| **`ToolPermissionHost`** | 由 Agent 服务或 CLI 注入：YAML 路径、热读快照、ACP、持久化、工作区、通道门控、场景钩子等。 |

可运行示例（仓库根目录）：

```bash
uv run python examples/permissions/permission_demo.py
```

源码入口：`openjiuwen.harness.security`（`AGENTS.md` 中亦有说明）。

---

## 2. 如何开启（DeepAgent）

同时满足：

1. **`DeepAgentConfig.permissions`** 为 `dict`，且 **`enabled: true`**。  
2. 通过 **`create_deep_agent(..., permissions={...}, permission_host=...)`** 传入（`permission_host` 可选但生产环境强烈建议配置）。

`DeepAgent` 在 **`_queue_pending_rails`** 中会调用 **`build_permission_interrupt_rail`**；仅当 `permissions.get("enabled")` 为真时才会挂载 **`PermissionInterruptRail`**。

最小 `permissions` 示例（由 `tiered_policy.evaluate_tiered_policy` 评估；`schema` 可选，仅作文档约定）：

```python
permissions = {
    "enabled": True,
    "schema": "tiered_policy",
    "permission_mode": "normal",
    "tools": {
        "read_file": "ask",
        "write_file": "deny",
    },
    "defaults": {"*": "allow"},
    "rules": [],
    "approval_overrides": [],
}
```

引擎不再提供与上述分层模型并行的旧式「纯 patterns」判定路径；配置应使用 `tools` / `defaults` / `rules` / `approval_overrides` 等字段表达策略。

## 3. `ToolPermissionHost` 字段说明

定义见 **`openjiuwen.harness.security.host.ToolPermissionHost`**。

| 字段 | 含义 |
|------|------|
| **`get_permissions_snapshot`** | 返回与配置里 **`permissions`** 同结构的 `dict`；在工具调用前若存在，Rail 会 **`update_config`** 与磁盘对齐。 |
| **`request_permission_confirmation`** | 对 **ASK** 异步征求确认；返回 **`PermissionConfirmResponse`**、字面量 **`"interrupt"`**（回退内置 Confirm）或 **`None`**（失败）。其中「总是允许」语义为 **`approved: true` 且 `auto_confirm: true`**（会走合并 ``permissions``、更新内存并写盘）；产品侧可实现 ACP 等协议；见 **`RequestPermissionConfirmationHook`**。 |
| **`persist_allow_rule`** | ``(permissions: dict) -> bool``：护栏已合并并 ``update_config`` 后调用，入参为整份新 ``permissions``；返回 ``False`` 会回滚内存。未设置时用 **`write_permissions_section_to_agent_config_yaml`** 写 YAML。 |
| **`resolve_workspace_dir`** | 外部路径校验用的工作区根目录。 |
| **`permission_yaml_path`** | **Agent 配置文件**路径；`persist_*` 写入该 YAML 根下的 **`permissions:`** 小节。文件可尚不存在，**父目录须存在**；首次写入时可用当前全局引擎配置引导航。 |
| **`tool_permission_checks_active`** | `Callable[[], bool]`：若返回 **False**，则跳过工具权限校验（等价放行）。**未设置**时始终执行校验；由宿主决定「当前上下文是否要做校验」（如按入口类型路由，实现可放在产品包）。 |
| **`permission_scene_hook`** | 在通用 tiered 判定前异步介入；返回 `None` 则继续走引擎。用于数字分身、owner_scopes 等产品逻辑（实现通常在宿主仓库）。 |

## 4. 读的是哪份「权限配置」？

有三条常见来源（可同时存在，职责不同）：

1. **启动时内存中的 `permissions`**  
   即传给 **`create_deep_agent(permissions=...)`** 的字典；来源由调用方决定（可从 YAML 读入后再传入）。

2. **运行时快照（可选）**  
   若配置了 **`get_permissions_snapshot`**，每次进入权限判定时会尝试用其返回值 **`update_config`**，避免仅更新了磁盘或全局单例引擎而 Rail 仍用旧配置。

3. **`PermissionEngine.config`**  
   引擎内部持有的当前策略；可能与上两者在热更新后合并或替换，以 Rail 内逻辑为准。

**说明**：openjiuwen **不**写死「必须从某个固定文件名读取」；路径由宿主通过 **`permission_yaml_path`** / 快照回调决定。

---

## 5. 写的是哪个文件？

持久化（如 **`approval_overrides`**、**`external_directory`** 白名单等）针对的是 **`permissions`** 整段策略（以 `dict` 形式合并后更新内存）。**落盘方式二选一**：

- **已设置 `ToolPermissionHost.persist_allow_rule`**：用户选择「总是允许」并合并成功后，护栏先 **`update_config`**，再**只**调用 **`persist_allow_rule(整份 permissions dict)`**；由宿主写入数据库 / 配置服务等。**不会**再调用 `write_permissions_section_to_agent_config_yaml`。回调返回 **`False`** 时，护栏会**回滚**内存中的 `PermissionEngine.config`。
- **未设置 `persist_allow_rule`**：使用内置 **`write_permissions_section_to_agent_config_yaml`**，把合并后的 **`permissions:`** 写进 **Agent 根配置文件**（YAML）的对应小节；**不是**单独名为 `permission_config` 的文件。

内置 YAML 写入时的路径解析（**`patterns._resolve_agent_config_yaml_path`**）：**仅**使用传入的 **`config_yaml_path`**（Rail 侧通常来自 **`ToolPermissionHost.permission_yaml_path`**）；未传入则无法落盘。**不依赖**环境变量解析该路径。

---

## 6. 宿主扩展与确认路径

- **`tool_permission_checks_active`**：由产品决定当前任务是否执行工具权限引擎；未注入则**始终**校验。  
- **`request_permission_confirmation`**：由产品实现 **ASK** 时的确认 UI（例如 **jiuwenclaw** 的 ACP `session/request_permission`）；返回 **`"interrupt"`** 时与未配置一样走内置 Confirm。  
- **`persist_allow_rule`**：由产品接管「总是允许」后的持久化（见上文第 5 节）；与 **`permission_yaml_path`** 二选一，**优先**走本回调。  
- **`permission_scene_hook`**：在 tiered 引擎之前返回 `("approve",)` / `("reject", msg)` 或 `None`；适合宿主独占的场景分支（实现可放在产品包，如 **jiuwenclaw** 的 `permission_scene_hook` + `owner_context`）。

## 7. 与全局安全说明的关系

- 通用安全风险（传输、SSRF、自定义组件等）：见 [安全配置](./安全配置.md)。  
- Guardrail 内容安全：见 [安全护栏 Guardrail](./高阶用法/安全护栏Guardrail.md)。  
- **工具权限（本文）**：控制**能否执行工具**及 **ASK 确认流**，与 SSL/SSRF 正交互补。
