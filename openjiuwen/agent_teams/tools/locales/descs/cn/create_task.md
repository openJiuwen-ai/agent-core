创建团队任务（仅 Leader 可用）。
**任务应聚焦于可交付成果和验收标准，不要规划具体执行步骤。**

## 使用场景

- 项目初始：批量创建完整的任务 DAG
- 执行过程中：补充新发现的任务
- 插入已有 DAG：设置 depended_by 创建反向依赖，将遗漏的前置任务插入已有依赖链

## 任务字段

- **title**: 简明描述任务目标（祈使语气，如 "实现用户认证"）
- **content**: 目标、验收标准和约束 — 不写具体操作步骤
- **task_id**（可选）: 自定义 ID，便于依赖引用（不提供则自动生成）
- **depends_on**（可选）: **"我依赖谁"** — 前置任务 ID 列表，须先完成才能开始本任务
- **depended_by**（可选）: **"谁依赖我"**（反向依赖）— 需要等待本任务完成的现有任务 ID 列表。列出的任务会自动变为 blocked 直到本任务完成

所有任务初始状态为 `pending`（有未解决依赖的为 `blocked`）。

## Tips

- 单个任务也用数组包裹 — tasks 始终是数组
- 描述目标，不描述步骤：content 写目标、验收标准、技术约束
- 单人认领：每个任务对应一个可独立交付的成果
- 用 depends_on / depended_by 构建依赖 DAG

## 粒度范例

以"用户认证"为例：

- **太细**：拆成"设计 User 表"、"实现 POST /login"、"实现 JWT 签发"、"写单测"——每个都是执行步骤而非可交付成果，验收只能描述动作
- **合适**：一个任务"实现用户登录能力（注册 + 登录 + 会话保持）"，content 写清目标、验收标准（API 完成注册/登录/续期）、约束（bcrypt + JWT）。单人可独立交付，按 API 行为验收
- **太粗**：一个任务"开发整个用户模块"——覆盖过宽，多人并行还要二次拆分；单人做成本高、验收条目模糊

## 强制流程

1. **创建前**：必须先调用 `view_task` 查看当前任务看板，避免重复创建、避免漏掉依赖、了解可复用的任务 ID
2. **创建后通知成员前**：再次调用 `view_task` 复查刚刚的写入是否符合预期（标题、依赖关系是否正确），确认无误后再 `send_message` 知会成员，避免把错误的任务广播出去
