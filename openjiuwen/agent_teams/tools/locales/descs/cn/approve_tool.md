审批或拒绝成员当前待审批的工具调用。Leader 调用后，系统会将审批结果发送给对应成员；若批准，成员可继续处理该工具调用；若拒绝，成员会收到反馈并需要改用其他方式继续。

这类请求通常出现在工具调用被安全策略、rail 或其他审批机制拦下时。

| 参数 | 用法 |
|---|---|
| **member_name** | 发起该工具审批请求的成员 member_name，应指向当前团队中的有效成员 |
| **tool_call_id** | 待恢复的中断 tool_call_id，必须对应这次审批请求中的具体工具调用 |
| **approved** | 是否批准这次工具调用。true 表示允许继续，false 表示拒绝并要求成员调整方案 |
| **feedback** | 审批反馈。拒绝时应说明原因和替代方向；批准时也可补充边界、风险提醒或使用约束 |
| **auto_confirm** | 是否对后续同名工具自动批准。仅在你明确接受该类工具后续继续使用时设置为 true |

适用场景：
- 成员的工具调用被审批机制拦下，需要 Leader 决定是否放行当前请求
- 工具本身可以继续使用，但需要附加边界、范围或安全提醒
- 当前工具调用不应继续，成员需要改用更安全或更合适的方案

注意：
- 批准的是这次待审批的具体工具调用，不等于成员整体任务已经通过审核
- 拒绝时不要只说"不行"，应给出可执行的修改方向或替代方案
- auto_confirm=true 会放宽后续同名工具的审批门槛，不应为了省事默认开启
